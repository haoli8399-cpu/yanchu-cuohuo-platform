## 变更前必须确认的三件事

每次改代码前，先回答这三个问题：

| # | 问题 | 答案示例 |
|:-:|------|---------|
| 1 | **这是哪个子项目？** | miniprogram / frontend-agent / frontend-admin / backend |
| 2 | **这改了哪个角色的功能？** | 活动公司 / 演员 / 运营 |
| 3 | **PRD 原始设计意图是什么？** | 翻 PRD 找对应的功能编号，确认原来怎么设计的 |

答不出来不准动手。

## 变更后必须做的验证

| 步骤 | 动作 | 说明 |
|:----:|------|------|
| 1 | 编译验证 | 改了什么项目就在哪个项目跑构建，不改的项目也得检查不破坏构建 |

| 3 | git 提交 | `git add -A && git commit -m "xxx" && git push` ，不要 rebase |

```
┌──────────────────────────────────────────────────┐

├──────────────────────────────────────────────────┤
```

## 历史上犯过的错（引以为戒）

| 错误 | 后果 | 根因 |
|------|------|------|
| 把前端Web端当小程序改 | 写React代码却以为在写小程序 | 没确认子项目类型就动手 |
| 花时间做设计稿代替代码 | 做了17个HTML页面，小程序一行没变 | 把文档当交付物 |
| 不理解业务模式直接开干 | 做成了招投标模式（演员报价竞标），和产品设计完全相反 | 没读 PRD 和业务流文档 |
| 在不理解全景时自己新建React项目 | 做了8个页面发现跟已有Web端功能重叠，全白做 | 没读项目全景图 |
| git rebase 搞乱提交历史 | 仓库状态异常 | 没按规范提交 |

**所有错误的根因只有一个：没读文档就动手。** 治理文件和这个8步流程就是为了解决这个问题。

## 与豪哥的沟通方式

- 豪哥风格：结论先行、直接批评
- 汇报格式：表格/列表/层级结构，10秒抓要点
- 不要大段纯文字
- 回答问题先给结论，再给理由
- 每次汇报完主动给3条"接下来可以做什么"的建议
- 分析方案时主动识别潜在风险
- 用大白话，不说术语
