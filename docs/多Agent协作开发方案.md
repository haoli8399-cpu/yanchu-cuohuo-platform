## 一、Agent 角色定义

```
┌─────────────────────────────────────────────────────────┐
│                  Agent-Orchestrator（主控）               │
│                  职责：调度 / 审查 / 部署 / 验证            │
│                  本窗口运行，不写业务代码                   │
└──────┬──────────────┬──────────────┬────────────────────┘
       │              │              │
       ▼              ▼              ▼
┌────────────┐ ┌────────────┐ ┌────────────┐
│Agent-Backend│ │ Agent-Web  │ │Agent-MiniP│
│  后端开发    │ │  Web前端   │ │  小程序    │
│  Schema/API │ │  运营后台+  │ │  双端开发   │
│  数据库/部署 │ │  活动公司端 │ │            │
└────────────┘ └────────────┘ └────────────┘
```

| Agent | 职责范围 | 技术栈 | 产物 |
|-------|---------|--------|------|
| **Orchestrator**（本窗口） | 任务拆解→派发→代码审查→部署验证→文档更新 | — | 无代码，管流程 |
| **Agent-Backend** | PostgreSQL Schema、API 实现、Supabase 部署、MeiliSearch 迁移、DeepSeek 集成 | Node.js + PostgreSQL + Supabase | Schema / API / 数据库迁移脚本 |
| **Agent-Web** | 运营后台 34 项 + 活动公司工作台 17 项 | Ant Design Pro + React | Web 前端 |
| **Agent-MiniProgram** | 甲方/活动公司小程序 13 项 + 演员小程序 14 项 | uni-app + NutUI | 小程序前端 |

## 二、协作流程（含代码审查）

```
每一轮迭代：

  Orchestrator 拆解任务
    │
    ├── 1. 读取相关文档（API_CONTRACT.md / Schema / 已有代码）
    ├── 2. 派发任务给 1-2 个 Agent（delegate_task，最大并发2）
    │
    ▼
  子 Agent 并行开发
    │
    ├── 读取 API_CONTRACT.md 对齐接口
    ├── 写代码
    ├── 自测通过
    └── 返回结果（代码路径 + 变更摘要 + 自测结果）
    │
    ▼
  Orchestrator 代码审查
    │
    ├── 检查项：
    │   ├── [ ] 接口格式是否对齐 API_CONTRACT.md
    │   ├── [ ] SQL 是否有注入风险（参数化查询）
    │   ├── [ ] 错误处理是否完整
    │   ├── [ ] 类型定义是否准确
    │   ├── [ ] 边界条件是否覆盖
    │   └── [ ] 代码风格是否一致
    │
    ├── 通过 → 进入部署验证
    └── 不通过 → 返回子 Agent 修改（附审查意见）
    │
    ▼
  Orchestrator 部署验证
    │
    ├── 后端：部署到腾讯云 → 跑 API 测试 → 检查日志
    ├── 前端：构建 → 本地预览 → 功能测试
    └── 通过 → 标记任务完成 + commit + push
    │
    ▼
  Orchestrator 更新文档
    │
    ├── 更新 working_deployment.md（部署状态）
    ├── 更新任务清单状态
    └── 向豪哥汇报进度
```

## 三、代码审查标准

### 3.1 后端审查清单

| # | 检查项 | 标准 |
|:-:|--------|------|
| 1 | **API 契约对齐** | 请求/响应格式与 API_CONTRACT.md 一致 |
| 2 | **SQL 注入防护** | 所有数据库操作使用参数化查询（`$1, $2`），禁止字符串拼接 |
| 3 | **权限校验** | 每个 API 端点检查用户角色（RBAC） |
| 4 | **错误处理** | 统一错误码 + 中文错误信息 |
| 5 | **输入校验** | 所有用户输入经过类型+长度+格式校验 |
| 6 | **事务完整性** | 涉及多表操作使用数据库事务 |
| 7 | **日志记录** | 关键操作记入操作日志表 |

### 3.2 前端审查清单

| # | 检查项 | 标准 |
|:-:|--------|------|
| 1 | **API 契约对齐** | 调用的 API 路径/参数/响应处理与契约一致 |
| 2 | **触控目标** | 可点击元素 ≥ 44px（对标微信规范） |
| 3 | **加载/空/错误状态** | 三种状态都有 UI 处理 |
| 4 | **错误处理** | API 报错时用户看到友好提示，不白屏 |
| 5 | **按钮位置** | 底部操作栏贴近拇指区域（小程序） |
| 6 | **代码风格** | 与项目已有代码风格一致 |

### 3.3 审查不通过的处理

```
审查意见 → 返回子 Agent → 「第X项不通过，问题：xxx，请修改后重新提交」
  → 子 Agent 修改 → 再次审查
  → 同一问题反复 3 次 → Orchestrator 自行修改并记录问题
```

## 四、任务派发规则

| 规则 | 说明 |
|------|------|
| **最多 2 并发** | 一次最多派发 2 个子 Agent 并行工作 |
| **无依赖可并行** | 后端 Schema + 前端脚手架 可同时开工 |
| **有依赖串行** | Schema 完成 → API 才能开始 → 前端对接 API |
| **隔离上下文** | 每个子 Agent 传完整的 task context（文件路径/接口定义/约束条件） |
| **只返回摘要** | 子 Agent 的中途过程不回传，只回传最终结果 |

## 五、质量铁律

| # | 规则 |
|:-:|------|
| 1 | 不跳过代码审查 |
| 2 | 不跳过端到端验证 |
| 3 | 不在审查通过前 commit |
| 4 | 不让同一个 bug 出现两次（出 bug 后更新审查清单） |
| 5 | 每完成一个模块 → commit + push |
| 6 | 文档与代码同步更新 |
