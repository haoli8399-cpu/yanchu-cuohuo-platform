## 一、背景：项目定位

| 维度 | 现状 |
|:----|------|
| **方向** | B2B 撮合（活动公司↔演员匹配+出方案+保履约） |
| **阶段** | **迭代维护** — 小步快跑，改功能、修bug、小迭代 |
| **工作模式** | 1 主控窗口（Orchestrator）+ 按需 spawn Developer |
| **原方案（v2.2）已废弃** | "演事"路线 + 4 Agent 并行大开发模式 |

## 二、Agent 角色定义（v3.1 扩展为 5 角色）

```
┌─────────────────────────────────────────────┐
│            Orchestrator（本窗口）              │
│ 职责：拆任务 → 写 TASK.md → 审查 → 验收      │
│       只做调度/审查/写文档/汇报，不写业务代码   │
└──┬───────────┬──────────┬──────────┬────────┘
   │           │          │          │
   ▼           ▼          ▼          ▼
┌────────┐ ┌────────┐ ┌────────┐ ┌────────────┐
│Developer│ │ Tester │ │Analyst │ │ Researcher │
│(工作窗) │ │(全职)  │ │(按需)  │ │ (按需)     │
│写代码   │ │验证功能 │ │梳理需求 │ │技术调研    │
└────────┘ └────────┘ └────────┘ └────────────┘

┌─────────────────────────────────────────────┐
│            Reviewer（按需启用）                │
│ 职责：只看代码质量，不改代码                    │
│       跨端改动时启用，日常由 OC 兼任             │
└─────────────────────────────────────────────┘
```

| Agent | 定位 | 启用时机 |
|-------|------|---------|
| **Orchestrator** | 项目经理+架构师+审查员+文档管理员 | 总是 |
| **Developer** | 写代码的执行者 | 每次任务 |
| **Tester** (v3.1 新增) | 功能验证员，独立于 Developer | Developer 回报后，复杂任务 spawn |
| **Analyst** (v3.1 新增) | 需求分析师，梳理模糊需求 | 需求不清晰时 |
| **Researcher** (v3.1 新增) | 技术/竞品调研员 | 选型不确定时 |
| **Reviewer** | 代码质量审查 | 跨端改动 / 按需 |

## 三、核心原则

| # | 原则 | 说明 |
|:-:|------|------|
| 1 | **文档主写子读** | Orchestrator 独占写入中心文档，子 Agent 只能读不能写 |
| 2 | **TASK.md 派发** | 每次任务写入 TASK.md，不在对话中粘贴任务 |
| 3 | **审批前置** | 改代码前先向豪哥说明计划并获取同意 |
| 4 | **确认即执行** | 豪哥说"OK" = 立即执行，不二次确认 |
| 5 | **三层审查** | 格式校验 → 规范审查 → 逻辑审查，缺一不可 |
| 6 | **自测闭环** | Developer 写完必须自测通过再回报 |
| 7 | **循环门禁** | 同任务审查不通过 ≥3 次 → 标记 Blocked → 问豪哥 |
| 8 | **每批 ≤3 文件 + ≤1 依赖** | 控制变更范围 |
| 9 | **风险登记** | 每次遇到新类型问题 → 记入风险登记册 |

## 四、完整工作流

### Phase 1：Orchestrator 准备（本窗口）

```
① 豪哥确认需求
② OC 更新 PROJECT_CENTER.md（当前任务状态）
③ OC 写 TASK.md（目标 + 验收标准 + 文件范围 + 约束条件）
④ OC 写 ACTIVE_TASK.md（同步任务信息）
⑤ OC 通知豪哥 → 豪哥到工作窗口发"读根目录 TASK.md"
```

### Phase 2：Developer 执行（工作窗口）

```
① 读 PROJECT_CENTER.md → 全局状态（第一读）
② 读 TASK.md → 本轮任务（第二读）
③ 读 FREEZE.md + 相关文档 → 对齐
④ 写代码
⑤ 自测通过（后端 curl / 前端 build）
⑥ 回报（code_paths + change_summary + self_test + known_issues + blocker）
⑦ 阻塞立即回报，不硬撑
```

### Phase 3：Tester 验证（如启用）

```
Developer 回报通过 →
  OC 决定是否 spawn Tester
    ├── 简单任务（1个文件、1个bug）→ OC 兼任，跳过
    └── 复杂任务（多文件/多端/业务流）→ spawn Tester
          │
          ▼
┌────────────────────────────────────────┐
│  Tester 功能验证                        │
│  [ ] API 请求能通，返回格式正确           │
│  [ ] 数据库读写正常，无权限泄漏            │
│  [ ] 前端页面能加载，无白屏/500           │
│  [ ] 空数据处理（列表无数据正常显示）      │
│  [ ] 超长输入/非法字符处理               │
│  [ ] 无权限用户被正确拦截                 │
│                                         │
│  发现问题 → 写入 rev/rNN/test.md + 打回  │
│  全部通过 → 写入 rev/rNN/test.md         │
└──────────────────┬─────────────────────┘
                   │ 通过
                   ▼
                  进入审查
```

### Phase 4：Orchestrator 验收（本窗口）

```
┌────────────────────────────────────────┐
│  第一层：格式校验                        │
│  ★ 检查 Developer 返回是否包含所有必填字段  │
│    [ ] code_paths（不为空）              │
│    [ ] change_summary（有内容）           │
│    [ ] self_test.passed（必须为 true）    │
│    [ ] self_test.build_output（前端必填）  │
│    [ ] blocker（必须声明，无阻塞填 false）  │
│                                         │
│  缺任何一项 → 直接打回 Developer           │
└──────────────────┬─────────────────────┘
                   │ 通过
                   ▼
┌────────────────────────────────────────┐
│  第二层：规范审查                        │
│  grep 扫描：参数化查询/console/any       │
│  ls 检查：文件是否存在                   │
│  不过 → 打回 Developer（附审查意见）       │
└──────────────────┬─────────────────────┘
                   │ 通过
                   ▼
┌────────────────────────────────────────┐
│  第三层：逻辑审查                        │
│  读 1-2 个关键文件，审查实际业务内容       │
│  边界条件 / FREEZE 一致性 / 架构合理性    │
│  不过 → 打回 Developer                   │
└──────────────────┬─────────────────────┘
                   │ 通过
                   ▼
┌────────────────────────────────────────┐
│  构建验证                               │
│  后端：npm run typecheck / curl 测试     │
│  前端：npm run build → 检查输出           │
│  小程序：编译检查                        │
│  失败 → 打回 Developer                   │
└──────────────────┬─────────────────────┘
                   │ 通过
                   ▼
┌────────────────────────────────────────┐
│  完成流程                               │
│  ① 归档：拷贝 TASK.md → rev/rNN/ + 写  │
│     output.md + 更新 test.md（如有）     │
│  ② 更新开发任务清单（状态变更）             │
│  ③ 更新 5 个一致性文件                    │
│  ④ git add + commit + push              │
│  ⑤ 更新 ACCEPTANCE.md + LOOP_LOG.jsonl   │
│  ⑥ 向豪哥汇报（表格/列表，10秒抓要点）     │
│  ⑦ 给 3 条"接下来做什么"建议               │
└────────────────────────────────────────┘
```

## 五、并行派发（v3.1 新增）

> 当一个任务可拆成 ≥2 个无依赖的子任务时，可以并行执行，节省时间。

### 判断条件

| 可并行 | 不可并行 |
|:------:|:--------:|
| 改两个互不依赖的独立模块（如改演员表和改活动列表） | 后端没写完 API，前端不能调 |
| 修两个不同端无关的 bug（如改小程序+改运营后台各自独立） | A 模块的输出是 B 模块的输入 |
| 调研和实现代码可以同时做 | 同一个文件不能两个 Agent 同时改 |

### 操作方式

```
OC 判断 → 拆成 TASK-A.md + TASK-B.md（各自自包含上下文）
  → 豪哥到 窗口A 说"读根目录 TASK-A.md"
  → 豪哥到 窗口B 说"读根目录 TASK-B.md"
  → Developer A + Developer B 各自执行 → 各自自测
  → 回报 → OC 做集成测试 → 合并归档 rev/rNN/
  → 同步 5 文件 → git commit + push
```

## 六、Loop 工程化

### 一个 Loop 长这样

```
┌─────────┐
│  Loop 1  │  → 目标清晰 → 实现 → 自测 → 审查 → 通过 → Done ✅
├─────────┤
│  Loop 2  │  → 审查不通过 → 打回修改 → 再验证 → 通过 → Done ✅
├─────────┤
│  Loop 3  │  → 发现新问题 → 更新 TASK → 实现 → 自测 → 审查 → Done ✅
├─────────┤
│  Loop 4  │  → 审查不通过 → 打回 → 再不过 → 标记 Blocked ❌ → 问豪哥
└─────────┘
```

### 循环相关文件

| 文件 | 位置 | 内容 | 谁写 | 什么时候更新 |
|:----|:----:|------|:----:|:-----------:|
| **ACTIVE_TASK.md** | 根目录 | 当前做什么（目标 + 验收标准 + 文件范围） | OC | 每次开始任务前 |
| **TASK.md** | 根目录 | 派发给 Developer 的完整任务 | OC | 每次派发前 |
| **ACCEPTANCE.md** | 根目录 | 验收表（逐项打勾 ✅/❌） | OC ← Developer | 任务完成后 |
| **LOOP_LOG.jsonl** | 根目录 | 每次循环记录（执行记录/耗时/验证结果） | OC | 每次循环结束 |

### 停止门禁（不能无限改下去）

| 门禁 | 触发条件 | 操作 |
|:---:|---------|------|
| 🛑 **连续失败** | 同任务审查不通过 ≥3 次 | 停 → 标记 Blocked → 问豪哥是不是方向错了 |
| 🛑 **预算耗尽** | 最多 5 次循环 / 60 分钟 | 停 → 标记 Done with Risk → 汇报还剩什么没搞完 |
| 🛑 **不可逆操作** | 改数据库/删文件/force push | 停 → 等豪哥确认 |
| 🛑 **方向冲突** | Developer 发现和 FREEZE/PROJECT_CENTER 矛盾 | 停 → 标记 Blocked → 等豪哥决策 |
| 🛑 **密钥/生产数据** | 需要 API Key / 账号 / 生产库 | 停 → 豪哥自己搞 |

### 完成判定

| 结论 | 含义 | 条件 |
|:---:|------|:----:|
| ✅ **Done** | 完美收官 | 所有验收项通过 + 构建通过 + 文档同步 + git push |
| ⚠️ **Done with Risk** | 核心功能好了，有边角没收 | 核心验收项通过 + 已知问题已记录 + 豪哥知情 |
| ❌ **Blocked** | 搞不下去了 | 方向冲突 / 需要豪哥决策 / 连续失败 ≥3 次 |
| 🔄 **Continue** | 搞了一轮还没完 | 有进度信号（新测试通过 / bug 范围缩小） |

## 七、子 Agent 启动流程（v3.1 简化链路）

所有子 Agent 在接到任务后、写代码前，必须按以下顺序阅读：

```
Step 1: PROJECT_CENTER.md    → 全局状态（第一读，3秒）
Step 2: TASK.md              → 本轮任务（第二读，含上下文约束
                               大部分情况不需要再跳读 FREEZE.md）
Step 3: 与任务相关的业务/技术文档  → 按需
Step 4: AGENT_RULES.md       → 遵守角色规范（必要时读）
```

> **注意**：TASK.md 已自带 context section 包含 FREEZE 约束。
> 除非标注 `⚠️ 需读 FREEZE.md`，否则跳过 FREEZE.md，省掉 1-2 步跳转。

## 八、审查标准

### 第一层：格式校验

| # | 检查项 | 检查方式 |
|:-:|--------|:--------:|
| 1 | code_paths（不为空） | 直接看 |
| 2 | change_summary（有内容） | 直接看 |
| 3 | self_test.passed === true | 直接看 |
| 4 | self_test.build_output（前端必填） | 直接看 |
| 5 | blocker 字段声明 | 直接看 |

### 第二层：规范审查

| # | 检查项 | 检查方式 |
|:-:|--------|:--------:|
| 1 | SQL 是否参数化 | grep $1 / 字符串拼接 |
| 2 | TypeScript `any` 使用 | grep any |
| 3 | console.log / debugger | grep |
| 4 | 文件是否存在 | ls 检查 |
| 5 | 前端：检查 index.html 标题 | grep '<title>' |

### 第三层：逻辑审查

| # | 检查项 | 检查方式 |
|:-:|--------|:--------:|
| 1 | 架构是否合理 | 审查 |
| 2 | 是否违反 FREEZE 决策 | 对照 FREEZE.md |
| 3 | 边界条件是否覆盖 | 读 1-2 个关键文件 |
| 4 | 三态（loading/empty/error） | 前端审查 |

## 十、风险登记册

### 已知风险

| # | 风险 | 影响 | 发生次数 | 缓解措施 |
|:-:|------|:--:|:----:|------|
| R1 | Agent 产出"表面正确、内容错误" | 页面内容完全偏离需求 | 1 | 构建验证 + 内容抽查 + 逻辑审查 |
| R2 | Agent 产出"描述未产出" | 报告说完成了但文件没写入 | 3 | 第一层审查 ls 检查 |
| R3 | 服务器内存不足（2C2G） | API 崩溃、SSH 超时 | 多次 | 精简部署、计划升 2C4G |
| R4 | Git push 超时 | 网络波动导致推送失败 | 多次 | http.version=HTTP/1.1、重试 |
| R5 | 子 Agent UUID 格式错误 | 种子数据无法导入 | 1 | 格式校验 |
| R6 | Agent 返回格式不完整 | OC 无法判断任务完成 | 3 | 格式校验前置 |
| R7 | macOS SIGTRAP（tsc 崩溃） | 本地类型检查不可靠 | 持续 | 不依赖本地 tsc 作为证据 |

### 潜在风险

| # | 风险 | 影响 | 预防措施 |
|:-:|------|:--:|------|
| P1 | 子 Agent 修改 docs/ 目录 | 文档被覆盖 | 子 Agent 规则禁止 + OC 审查 |
| P2 | 方向漂移（Agent 跑偏） | 做了一半发现方向不对 | 启动必读 3 文档 + 循环门禁 |
| P3 | 多人 IDEA 窗口冲突 | 同时改同一文件 | 一次只 spawn 1 个 Developer |
