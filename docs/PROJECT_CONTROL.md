## 二、子 Agent 工作规范（SOP）

### 2.1 启动前：必须阅读的文档（按顺序）

```
每个子 Agent 在接到任务后、写代码前，必须按以下顺序阅读：

1. FREEZE.md          → 知道哪些决策不可更改
2. 开发任务清单.md      → 了解当前任务在整体中的位置
3. 与任务相关的业务文档   → 理解业务逻辑
   - 涉及演员 → 演员分层管理体系.md + 演员信誉分设计方案.md
   - 涉及订单 → 完整业务流.md
   - 涉及功能 → 产品功能清单.md
4. API_CONTRACT.md     → 对齐接口格式（如涉及 API）
5. 技术方案文档.md      → 了解技术约束（如涉及后端）
6. CODE_STANDARDS.md   → 遵守代码规范
```

### 2.2 工作中：必须遵守的规则

| # | 规则 | 违规后果 |
|:-:|------|---------|
| 1 | **先读 API_CONTRACT.md 再写接口** | 审查不通过，打回重写 |
| 2 | **不修改 FREEZE.md 中的决策** | 审查不通过，警告 |
| 3 | **不确定的业务逻辑 → 标注 `TODO: OC_CONFIRM`** | 自行猜测导致返工 |
| 4 | **所有数据库操作使用参数化查询** | 审查不通过 |
| 5 | **所有 API 端点检查权限** | 审查不通过 |
| 6 | **写完自测通过再提交** | 审查不通过 |

### 2.3 结束后：必须提交的产物

```yaml
# 子 Agent 任务完成时返回的标准格式
result:
  code_paths:           # 变更的文件路径列表
    - "backend/src/api/skus.ts"
    - "backend/src/schema/skus.sql"
  change_summary:       # 一句话描述做了什么
    "实现 SKU CRUD API：新增/编辑/列表/上下架 4 个端点"
  self_test:            # 自测结果
    passed: true
    notes: "本地测试通过，4 个端点均返回正确数据"
  docs_to_update:       # 因本次变更需要同步更新的文档
    - "API_CONTRACT.md  # 新增了 /v1/skus 端点定义"
    - "working_deployment.md  # 新增了 skus 表迁移"
  known_issues:         # 已知问题（如有）
    - "上下架状态的权限校验待 P-32 RBAC 完成后补充"
```

## 三、Orchestrator 工作规范（SOP）

### 3.1 每轮开始

```
1. 读取 开发任务清单.md → 确定本轮要执行的 1-2 个任务
2. 检查依赖：前置任务是否已完成？
3. 构建子 Agent context（任务描述 + 相关文档路径 + 约束条件）
4. delegate_task 派发
```

### 3.2 每轮结束

```
1. 检查子 Agent 返回结果中的 blocker 标记 → 有阻塞优先处理
2. 分层审查：
   ├── 第一层 OC 审规范（机械检查：SQL参数化/类型/格式，5min）
   ├── 第二层 Agent 交叉审逻辑（BE审Web/Web审MP，10min）
   └── 第三层 OC 终审（架构/FREEZE一致性/边界条件，5min）
3. 审查通过 → 集成测试（前后端联调验证 API 契约一致性）
4. 集成测试通过 → 部署验证
5. 检查 DoD（6项全部满足才算完成）：
   ├── [ ] 三层审查通过
   ├── [ ] 集成测试通过
   ├── [ ] 部署验证通过
   ├── [ ] 开发任务清单.md 已更新
   ├── [ ] API_CONTRACT.md 已同步（如有变更）
   └── [ ] commit + push
6. 向豪哥汇报进度（附完整任务清单）
```

### 3.3 文档更新铁律

| 规则 | 说明 |
|------|------|
| **主写子读** | Orchestrator 负责写文档，子 Agent 只读不写 |
| **即写即更** | 任务完成后立即更新，不攒批 |
| **先更后推** | 更新文档 → commit → push，三者同步 |
| **出错追责在 OC** | 文档落后于代码，责任在 Orchestrator |

## 四、冻结文件（FREEZE.md）

> FREEZE.md 记录所有已确认、不可再讨论的决策。
> 子 Agent 读取后，不得质疑或修改这些决策。
> 如需修改冻结决策，必须由豪哥确认后，Orchestrator 更新 FREEZE.md。

### FREEZE.md 内容结构

```markdown
# 冻结决策文件

> 以下决策已经豪哥确认，子 Agent 不得质疑或自行修改。
> 如需修改，必须先经豪哥确认 → Orchestrator 更新本文档。

## 1. 商业模式
- 平台定位：B2B 演出供应链保障平台（批发商模式）
- 4 个角色：平台运营 / 活动公司 / 演员 / 甲方（第一阶段暂不开放）

## 2. 价格体系
- 内部结算价：T0-T6 咖位定价（详见演员分层管理体系）
- 活动公司价：甲方标准价 × 0.7
- 甲方标准价：外出演出 6000/10000/12000/16000...

## 3. 技术栈
- 后端：Node.js Fastify（非 Supabase Edge Functions）
- 数据库：PostgreSQL（Supabase 自托管）
- 搜索：MVP 用 PostgreSQL 内置，不上 MeiliSearch
- 状态机：后端配置对象，不用 XState

## 4. 产品决策
- 不做在线支付 → 线下登记
- 不做竞标模式
- 活动公司两步入驻（注册即用，交易前认证）
- 第一阶段不开放甲方端
...
```

## 五、代码规范标准（CODE_STANDARDS.md）

> 所有 Agent 必须遵守的最低代码标准。Orchestrator 按此标准审查。

### 5.1 后端（Node.js + PostgreSQL）

| # | 规范 | 说明 |
|:-:|------|------|
| 1 | TypeScript 严格模式 | 禁止 `any`，所有函数有返回类型 |
| 2 | 参数化查询 | 禁止字符串拼接 SQL，全部用 `$1, $2` |
| 3 | 统一错误格式 | `{ code: number, message: string, data: null }` |
| 4 | 权限中间件 | 每个端点声明所需角色，中间件统一校验 |
| 5 | 输入校验 | 使用 zod 或 joi，不信任任何用户输入 |
| 6 | 事务包裹 | 涉及多表写操作使用 `BEGIN/COMMIT/ROLLBACK` |
| 7 | 操作日志 | 所有 CUD 操作写入 `operation_logs` 表 |
| 8 | 文件命名 | `kebab-case.ts`，路由文件与 API 路径对应 |

### 5.2 前端（React + Ant Design Pro）

| # | 规范 | 说明 |
|:-:|------|------|
| 1 | TypeScript 严格模式 | 禁止 `any` |
| 2 | 触控目标 ≥ 44px | 对标微信设计规范 |
| 3 | 三态处理 | 每个数据组件处理 loading / empty / error |
| 4 | API 调用统一 | 全部通过 `apiClient.ts`，不直接调 fetch/axios |
| 5 | 组件拆分 | 单文件 ≤ 300 行，超过则拆分 |
| 6 | 微信风格优先 | 底部操作栏、音浪动画上方按钮下方 |

### 5.3 小程序（uni-app + NutUI）

| # | 规范 | 说明 |
|:-:|------|------|
| 1 | 同前端 1-5 条 | 全部适用 |
| 2 | 触控目标 ≥ 44px | 底部操作栏贴近拇指区域 |
| 3 | 振动反馈 | 关键操作（确认/签到）触发 `wx.vibrateShort()` |
| 4 | 分包加载 | 单包 ≤ 2MB，主包只放首页+登录 |

### 5.4 通用

| # | 规范 |
|:-:|------|
| 1 | 禁止硬编码配置（价格/URL/密钥），全部走环境变量或配置表 |
| 2 | 注释用中文 |
| 3 | commit message 格式：`类型: 简短描述`（如 `feat: SKU CRUD API`） |
| 4 | 禁止提交 `console.log` 到主分支 |
