39|## 1. 产品战略定位
40|
41|### 1.1 竞争策略
42|
43|**喜剧工厂是「AI驱动的商演交易平台」**，不是纯撮合平台，也不是纯AI工具。
44|
45|| 维度 | 撮合平台 | AI经纪人工具 | 喜剧工厂（融合模式） |
46||:----:|:--------:|:-----------:|:-------------------:|
47|| 供给端 | 需要地推 | 不需要 | ✅ 已有艺人/SKU库 |
48|| 需求端 | 需要获客 | 获客成本高 | ✅ 已有活动公司 |
49|| 交易效率 | 人工匹配 | AI自动 | ✅ AI辅助+人工确认 |
50|| 壁垒 | 网络效应 | 数据+AI | ✅ 网络+AI双重壁垒 |
51|| 规模化 | 重运营 | 轻模式 | ✅ 逐步降低运营成本 |
52|
53|### 1.2 核心差异化
54|
55|> **不是帮用户「管理演出」，而是帮用户「把商演需求变成成交订单」。**
56|
57|- 竞争对手（活动行、大麦）：演出信息平台，交易在站外
58|- 我们：商演交易平台，从询价到签约到收款全闭环
59|- 终局：AI经纪人接管80%标准化工作，人做20%复杂决策

### 4.3 小程序 vs PC Web 的差异化定位

| 场景 | 小程序（主力） | PC Web（补充） |
|:----|:-----------|:------------|
| 快速浏览 | ✅ 手势操作、微信登录 | — |
| 提交需求 | ✅ AI对话+快速表单 | ⚠️ 复杂需求时可用 |
| 深度方案比价 | — | ✅ 多列对比、筛选器 |
| 查看/跟进订单 | ✅ 消息推送 | ⚠️ 查看 |
| 审阅/签署合同 | — | ✅ 大屏阅读、下载打印 |
| 数据复盘 | — | ✅ 历史成交数据查看 |

**策略：** 小程序是「快速决策+看状态」的移动入口，PC Web 是「深度对比+合同管理」的工作站。两个端功能集有差异，不追求一一对应。

## 6. 核心业务流程

### 6.1 活动公司端（完整链路）
281|
282|```
283|发现方案 → 查看详情 → 提交需求 → 收到报价 → 确认方案 → 
284|签约 → 等待演出 → 演出执行 → 评价
285|```
286|
287|| 步骤 | 页面 | 功能 |
288||:----:|:----:|------|
289|| 1 | 发现页 `discover` | 搜索/Banner/分类/推荐 |
290|| 2 | SKU详情 `sku/detail` | 默认配置+价格+介绍+场地+案例+评价 |
291|| 3 | 提交需求 `request/submit` | 自定义配置器（级别/时长/人数）+ 实时报价 |
292|| 4 | 需求列表 `request/list` | 状态跟踪 |
293|| 5 | 需求详情 `request/detail` | AI方案+确认/不采纳+评价 |
294|| 6 | 个人中心 `user` | 收藏/订单/企业信息/设置/通知 |
295|

### 6.4 商机状态机

> 核心原则：每个需求都是一条商机，从进入到关闭必须经过明确的状态流转。运营人员根据状态执行对应动作，系统根据状态自动触发提醒。

**商机状态流转：**

```
                    ┌─────────────┐
                    │    new      │  ← 新需求到达
                    │   新需求     │
                    └──────┬──────┘
                           │ AI解析+人工审核
                           ▼
                    ┌─────────────┐
              ┌─────│ qualified   │  ← 需求有效，进入报价
              │     │   有效      │
              │     └──────┬──────┘
              │            │ 运营生成报价
              │            ▼
              │     ┌─────────────┐
              │     │   quoted    │  ← 报价已发送
              │     │  已报价     │
              │     └──────┬──────┘
              │            │ 客户有异议/还价
              │            ▼
              │     ┌─────────────┐
              │     │ negotiating │  ← 谈判中
              │     │  谈判中     │
              │     └──┬──────┬──┘
              │        │      │ 等待客户决策
              │        │      ▼
              │        │ ┌──────────────┐
              │        │ │pending_client│ ← 等客户
              │        │ │  等客户      │
              │        │ └──┬───────┬──┘
              │        │    │       │
              │        ▼    ▼       ▼
              │     ┌──────────┐ ┌──────────┐
              │     │   won    │ │   lost   │
              │     │  成交     │ │  丢单    │
              │     └──────────┘ └──────────┘
              │
              └──  new → lost（无效需求直接关闭）
```

**各状态对应运营动作：**

| 状态 | 运营动作 | 自动提醒 |
|:-----|:--------|:--------|
| **new** | AI解析需求、人工确认信息完整性 | 新需求到达即时通知 |
| **qualified** | 确认需求有效、选择SKU模板、匹配艺人 | 2h内未处理→升级提醒 |
| **quoted** | 报价已发送、等待客户查看 | 24h未读→提醒运营；48h未回复→AI生成跟进话术 |
| **negotiating** | 客户有异议、调整方案/价格 | 每次沟通后24h无进展→提醒 |
| **pending_client** | 等待客户最终决策 | 48h无动静→AI生成温和提醒话术 |
| **won** | 进入签约+排期+演出执行流程 | 自动归档 |
| **lost** | 记录丢单原因、分析复盘 | 周报汇总丢单分析 |

**丢单原因（必须记录）：**

| 原因分类 | 说明 |
|:--------|:----|
| 预算太低 | 客户预算无法覆盖最低方案 |
| 日期不合适 | 所有艺人档期已满 |
| 客户无回应 | 报价后客户失联 |
| 被竞品 | 客户选择了其他供应商 |
| 方案不匹配 | 没有合适类型的方案 |
| 客户取消活动 | 活动本身取消 |
| 价格太高 | 客户认为价格超出预期 |
| 非真实需求 | 询价但实际无购买意向 |

> **设计原则：** 丢单和成交同样重要——不知道为什么丢单，就无法优化成交效率。Phase 1 开始就必须记录丢单原因，不可省略。

### 6.5 报价版本管理

> 报价不是一次定稿的。客户嫌贵就降配、追加需求就升级方案——每次调整都应记录为一个版本，可追溯、可分析。

**报价版本示例：**

| 版本 | 方案 | 价格 | 状态 | 备注 |
|:----:|------|:----:|:----:|------|
| v1 | T3 脱口秀 60min 2人 | ¥6,000 | 已发送 | 初始报价 |
| v2 | T4 脱口秀 45min 2人 | ¥4,500 | 已发送 | 客户嫌贵，降配 |
| v3 | T3 脱口秀+互动 90min 3人 | ¥8,000 | 已确认 | 客户加互动需求 |

**报价版本数据模型（quotes 表）：**

| 字段 | 类型 | 说明 |
|:-----|:----|:----|
| `id` | UUID | 报价单ID |
| `demand_id` | UUID | 关联需求 |
| `opportunity_id` | UUID | 关联商机 |
| `version` | int | 版本号（v1, v2, v3...） |
| `total_price` | decimal | 总价（甲方标准价） |
| `channel_price` | decimal | 活动公司渠道价 |
| `cost_price` | decimal | 艺人成本（内部） |
| `margin` | decimal | 毛利（total - cost） |
| `valid_until` | timestamp | 报价有效期 |
| `status` | enum | draft/sent/viewed/confirmed/rejected/expired |
| `rejection_reason` | text | 客户拒绝原因 |
| `ai_generated` | boolean | 是否AI生成 |
| `operator_modified` | boolean | 运营是否手动修改过 |
| `created_at` | timestamp | 创建时间 |

**版本管理规则：**
- 同一需求可以有多个报价版本
- 客户每次反馈（嫌贵/改需求）→ 创建新版本
- 旧版本保留历史记录，不可删除
- 最终确认的版本标记为 `confirmed`
- 报价对比：可查看 v1→v2→v3 的变化轨迹（做了什么调整、为什么）

### 6.6 争议处理流程（Phase 2+ 系统化）

> **Phase 1：** 采用人工 SOP + 记录（在 follow_up_logs 和 dispute_records 中记录）。  
> **Phase 2+：** 推进系统化争议裁决流程如下。

演出服务具有不可逆性，线下问题（质量、迟到、换人）无法自动判定，需建立标准化争议裁决机制。

**争议类型与赔付标准：**

| 争议类型 | 判定方式 | 赔付方案 |
|:--------|:--------|:--------|
| 艺人迟到 | 签到时间对比演出时间 | 迟到30分钟以上→减免20%费用 |
| 演出缩水 | 现场协调人确认 | 按比例退款 |
| 临时换人 | 未获甲方同意 | 减免30%费用 |
| 演出质量严重不符 | 甲方举证+平台仲裁 | 退款50-100% |
| 甲方无故取消 | 签约时间判断 | 24h内取消→定金不退；24h+→退还50% |

**争议 SOP：** 任一方发起→平台运营48h内仲裁→资金按裁决解冻→记录入 dispute_records 表→影响供应商信誉分。

1058|### 12.2 API 设计原则
1059|
1060|1. **RESTful：** 所有端点以 `/v1/` 开头
1061|2. **Zod 校验：** 所有请求参数使用 Zod Schema 校验
1062|3. **分页标准：** `page: 1-*, pageSize: 1-1000, 返回 paginatedResponse`
1063|4. **错误码统一：** `0` 成功 / `1xxx` 认证 / `99xx` 参数 / `9999` 服务器错误
1064|5. **JWT 认证：** 所有需要认证的端点使用 `authMiddleware`
1065|6. **API Contract：** 所有 API 变更必须同步更新 `docs/API_CONTRACT.md`
