## 1. 产品战略定位

### 1.1 竞争策略

**喜剧工厂是「AI驱动的商演交易平台」**，不是纯撮合平台，也不是纯AI工具。

| 维度 | 撮合平台 | AI经纪人工具 | 喜剧工厂（融合模式） |
|:----:|:--------:|:-----------:|:-------------------:|
| 供给端 | 需要地推 | 不需要 | ✅ 已有演员/SKU库 |
| 需求端 | 需要获客 | 获客成本高 | ✅ 已有活动公司 |
| 交易效率 | 人工匹配 | AI自动 | ✅ AI辅助+人工确认 |
| 壁垒 | 网络效应 | 数据+AI | ✅ 网络+AI双重壁垒 |
| 规模化 | 重运营 | 轻模式 | ✅ 逐步降低运营成本 |

### 1.2 核心差异化

> **不是帮用户「管理演出」，而是帮用户「把商演需求变成成交订单」。**

- 竞争对手（活动行、大麦）：演出信息平台，交易在站外
- 我们：商演交易平台，从询价到签约到收款全闭环
- 终局：AI经纪人接管80%标准化工作，人做20%复杂决策

## 2. 产品一句话定义

> **AI驱动的B2B商演交易平台——帮助活动公司15分钟完成从「有个需求」到「拿到报价方案」的全过程，帮助演员/厂牌从零散微信询价走向标准化、可跟进的订单管理。**

## 3. 产品不是什么

| 不是 | 为什么 |
|:----|:----|
| ❌ **不是纯招投标平台** | 活动公司不应该"几家公司竞价"，而是平台基于需求标准化匹配最优方案 |
| ❌ **不是大众点评/大麦网** | 核心用户是B端活动公司，不是C端消费者 |
| ❌ **不是艺人管理系统** | 演员管理是底层数据，不是产品主入口 |
| ❌ **不是全自动AI** | 首版坚持：AI建议+人确认，不做AI自动对外发送 |
| ❌ **不是全行业平台** | 只聚焦脱口秀/喜剧/小剧场商演，不做演唱会/明星经纪 |

## 4. 目标用户与核心场景

### 4.1 用户角色（7类）

| 角色 | 定义 | 核心需求 | 使用端 |
|:----:|------|---------|:------:|
| 🏢 活动公司 | 企业活动策划方、甲方品牌部、年会/团建负责人 | 快速找方案+比价+签约 | 小程序 + PC Web |
| 🎯 自营运营 | 喜剧工厂雇员 | 处理自营订单、管自营演员、看利润 | supplier-console (PC) |
| 🔧 平台管理员 | 平台方员工 | 供应商审核、佣金设置、全平台看板、仲裁 | platform-admin (PC) |
| 🏪 第三方经纪公司 | 入驻的演出经纪公司 | 接平台订单、管旗下演员、看佣金/利润 | supplier-console (PC) |
| 🎤 独立演员 | 个人入驻的演员 | 接单、管个人SKU、看收入 | supplier-console小程序版 + 演员小程序 |
| 🎭 演员（被管理） | 经纪公司或平台旗下的演员 | 看排期、签到、结算、信誉 | 演员小程序 |
| 👥 经纪公司艺人 | 第三方经纪公司旗下的演员 | 看排期、签到、结算、信誉 | 演员小程序 |

**角色关系说明：**

- 自营运营、第三方经纪公司、独立演员共用同一套 **supplier-console**（PC为主，小程序做核心操作补充）
- 平台管理员使用独立的 **platform-admin**
- 独立演员是特殊角色：既是 supplier（管理自己SKU）又是 performer（接单演出）
- 小程序提供两个角色入口：
  - **活动公司模式**：默认 tab 列（发现 | 方案 | 提需求 | 消息 | 我的）
  - **演员模式**：切换后显示排期/结算/签到 tab

### 4.2 核心用户场景

**场景一：活动公司年会找脱口秀（已有）**

```
HR打开小程序 → 
  方式A: 浏览SKU列表，点「脱口秀标准版」，看详情和默认价格，点「获取报价」
  方式B: 输入/粘贴需求文字"年会10月20日，300人，要脱口秀，预算1万以内"
        → AI推荐方案+估价 → 确认提交

提交后 → 运营工作台收到任务 → AI预生成方案+推荐演员 
       → 运营确认/调整 → 点击发送 → 活动公司微信收到方案通知
       → 活动公司确认 → 签约 → 付款 → 演出现场签到 → 评价
```

**场景二：演员管理排期（已有）**

```
演员打开小程序 → 看到今日排期 → 看到待确认任务 → 
确认演出 → 当天签到打卡 → 演出后进入结算流程 → 团队收到款 → 信誉分更新
```

**场景三：自营运营处理需求（新增）**

```
自营运营打开 supplier-console (PC) → 
工作台展示新需求卡片 + AI预生成方案 →
  查看AI推荐的SKU+演员+报价 →
  调整（换演员级别/改时长/调价格） →
  点击「发送方案」→ 活动公司收到通知 →
  跟踪确认状态 → 活动公司确认后安排演员 → 跟进演出执行
```

**场景四：第三方经纪公司接单（新增）**

```
第三方经纪公司打开 supplier-console (PC) → 
工作台收到平台分配的需求 →
  查看需求详情（人数/预算/日期/活动类型） →
  基于平台推荐报价，自主报价（可调整） →
  安排旗下演员（选人+定档期） →
  提交方案 → 活动公司端收到方案 →
  确认后签约 → 演出执行 → 结算 → 平台抽佣
```

**场景五：独立演员管理自己（新增）**

```
独立演员打开小程序，切换到「演员模式」→ 
  查看排期日历 + 待确认演出 →
  
同时打开 supplier-console 小程序版 →
  管理个人SKU（价格/时长/类型/可演出日期） →
  查看历史订单和收入 →
  收到平台订单 → 确认/拒绝 →
  演出 → 签到 → 结算 → 收入到账
```

## 6. 核心业务流程

### 6.1 活动公司端（完整链路）

```
发现方案 → 查看详情 → 提交需求 → 收到报价 → 确认方案 → 
签约 → 等待演出 → 演出执行 → 评价
```

| 步骤 | 页面 | 功能 |
|:----:|:----:|------|
| 1 | 发现页 `discover` | 搜索/Banner/分类/推荐 |
| 2 | SKU详情 `sku/detail` | 默认配置+价格+介绍+场地+案例+评价 |
| 3 | 提交需求 `request/submit` | 自定义配置器（级别/时长/人数）+ 实时报价 |
| 4 | 需求列表 `request/list` | 状态跟踪 |
| 5 | 需求详情 `request/detail` | AI方案+确认/不采纳+评价 |
| 6 | 个人中心 `user` | 收藏/订单/企业信息/设置/通知 |

### 6.2 演员端

```
收到排期 → 确认档期 → 现场签到 → 结算 → 信誉更新
```

### 6.3 运营端

```
新需求通知 → 查看需求 → AI辅助生成方案 → 调整/确认 → 发送
→ 跟踪确认 → 安排演员 → 跟进执行 → 完成归档
```

## 7. 商业模式

### 7.1 模型定位：京东自营模型

> 喜剧工厂借鉴京东自营模式——平台自己是市场管理者，"喜剧工厂自营"是平台上一家供应商。活动公司在方案列表中看到的是「混排」结果：自营方案和第三方供应商方案并列展示，通过供应商标识区分。

### 7.2 三层渐进收入模型

#### Tier 1：自营赚差价（当前阶段）

- **模式：** 喜剧工厂签约演员，作为唯一供应商
- **盈利方式：** 赚取演出差价（活动公司付¥6,000 → 演员出场费¥4,000 → 平台利润¥2,000）
- **目标毛利率：** 30-40%
- **启动条件：** 现在（已有演员库和SKU体系）
- **关键动作：** 定价策略 + 运营效率提升

#### Tier 2：平台抽佣（月需求≥100单后开放）

- **模式：** 第三方经纪公司/独立演员入驻，与自营混排展示
- **盈利方式：**
  - 平台带来的订单 → 抽佣 **10-15%**（按订单金额计算）
  - 第三方自己带来的订单 → **不抽佣**
- **目标佣金率：** 12%
- **启动条件：** 月需求≥100单（约 Phase 2 中后期）
- **关键动作：** 供应商入驻流程 + 佣金结算系统 + 品质管控

#### Tier 3：AI Agent 订阅（Agent 打磨成熟后开放）

- **模式：** 第三方经纪公司/独立演员可订阅 AI Agent 处理自有客户
- **盈利方式：**
  - 个人版：**¥199/月**（独立演员用）
  - 企业版：**¥499/月**（经纪公司用）
- **核心功能：** AI 报价、智能跟进、合同生成、自有客户 CRM
- **规则：** 订阅用户的自有客户订单 → 0 佣金（订阅费已覆盖）
- **启动条件：** AI Agent 功能成熟、供应商量达到 10+ 家
- **关键动作：** Agent 产品化 + 订阅计费系统 + 自有客户管理

### 7.3 三层收入关系

三层模型不是替代关系，而是叠加关系：

| 收入来源 | Tier 1 | Tier 2 | Tier 3 |
|:--------|:-----:|:-----:|:-----:|
| 自营差价 | ✅ | ✅ | ✅ |
| 平台佣金 | — | ✅ | ✅ |
| AI Agent 订阅 | — | — | ✅ |

**逐步演进示意：**

```
Tier 1 阶段（当前）
  收入 = 自营差价
  月营收目标：¥6万（假设月订单10-15单，均价¥6,000，毛利40%）

Tier 1+2 阶段（Phase 2 中后期）
  收入 = 自营差价 + 平台佣金
  自营：保持不变
  第三方：假设月订单20单，均价¥5,000，佣金12%
  月营收目标：自营¥6万 + 佣金¥1.2万 = ¥7.2万

Tier 1+2+3 阶段（Phase 3）
  收入 = 自营差价 + 平台佣金 + Agent订阅
  自营：¥6万
  第三方佣金：¥3万（月订单50单，均价¥5,000，佣金12%）
  Agent订阅：¥7,500（5家公司×¥499 + 20个演员×¥199）
  月营收目标：¥9.75万
```

### 7.4 价格体系概要

| 价格类型 | 说明 | 适用对象 |
|:--------|:----|:--------|
| 甲方标准价 | 活动公司端展示的对外报价 | 所有活动公司 |
| 活动公司渠道价 | 活动公司的成本价（= 标准价 × 0.7） | 自营订单 |
| 内部结算价 | 平台给演员的出场费 | 自营演员 |
| 第三方自主定价 | 第三方供应商自行定价 | 第三方订单 |
| Agent 订阅费 | 个人¥199/月，企业¥499/月 | Tier 3 订阅用户 |

#### 10.2.3 AI 主动追问机制

**必须追问，否则就不是AI经纪人。** 真实场景中活动公司HR通常只写一句模糊需求（"年会想搞个脱口秀"），不追问无法匹配方案。

**追问策略：**

```
用户: "年会想搞个脱口秀"
         ↓
AI: 好的！我先确认几个信息：
    ① 多少人参加？
      [50-100人] [100-300人] [300-500人] [500人以上]

用户: 选 100-300人
         ↓
AI: ② 预算范围？
      [5千以内] [5千-1万] [1万-2万] [2万以上] [还没想好]

用户: 1-2万
         ↓
AI: ③ 大概什么时间？
      [请选择日期]

用户: 选 10月20日
         ↓
AI: 明白了——300人年会，10月20日，预算1-2万。
    推荐「脱口秀标准版」T3级 60min ¥6,000 →
    再加一个备选「脱口秀旗舰版」T2级 90min ¥9,000 →
    [确认提交]
```

**追问规则：**

| 缺失字段 | 追问方式 | 是否必填 |
|:--------|:--------|:--------:|
| 人数 | 提供几个常见范围按钮（快速选择） | 是（影响演员配置） |
| 日期 | 日期选择器 | 是（影响档期匹配） |
| 预算 | 几个价格区间按钮 | 是（影响SKU推荐） |
| 地点 | 输入框 | 否（可选，辅助场地匹配） |
| 特殊要求 | 输入框 | 否（所有其他要求写在这里） |

**追问次数上限：最多3轮。3轮后使用已有信息直接匹配。** 避免把用户困在问卷里。

**追问状态指示：**

```
追问中（如"①/③ 还需补充2个信息"）
追问完成 → 自动开始匹配
匹配中（如"🤖 正在为你匹配方案..."）→ 展示结果卡片
```

#### 10.2.4 AI 自然语言识别

**输入：** 自由文本（上限500字）

**识别字段：**

| 字段 | 识别方式 | 示例 |
|:----|:--------|:----|
| 演出类型 | 关键词匹配 + 语义理解 | "脱口秀""即兴""魔术""喜剧" |
| 活动类型 | 上下文判断 | "年会""团建""发布会""开盘" |
| 人数规模 | 数字提取 | 300人 |
| 预算范围 | 数字+语义 | "1万以内""5千-8千" |
| 日期/时段 | 日期识别 | 10月20日/下个月初 |
| 地点 | 地名识别 | 上海浦东 |
| 特殊要求 | 语义理解 | "要女演员""最好会互动""不要太黄段子" |

#### 10.2.5 推荐匹配逻辑

```
输入文字 → AI 结构化 → 
  Step 1: 匹配演出类型 → 过滤SKU库
  Step 2: 匹配预算 → 过滤价格区间
  Step 3: 匹配人数 → 推荐演员人数
  Step 4: 匹配日期 → 检查档期
  Step 5: 历史参考 → 查询同类案例成交价
  → 输出 Top 2-3 方案
```

| **合同生成** | AI 基于方案信息自动生成合同草稿，供应商审阅后发送 | supplier-console → 一键生成 |
| **自有客户 CRM** | 供应商管理自己的客户（非平台来源），跟踪沟通记录和成交历史 | supplier-console 独立模块 |

### 11.5 在线签约 + 支付闭环

#### 11.5.1 签约流程

```
方案确认 → AI 生成合同草稿 →
  供应商审阅/编辑 →
  活动公司查看/签署 →
  供应商签署 →
  合同归档 → 进入支付流程
```

#### 11.5.2 支付流程

```
合同签署 → 活动公司支付定金（50%）→
  演员安排 → 演出执行 → 活动公司确认完成 →
  支付尾款（50%）→ 供应商结算
```

#### 11.5.3 资金流向

| 订单类型 | 资金流向 | 平台收费 |
|:--------|:--------|:--------|
| 自营订单 | 活动公司 → 平台 → 7天后 → 演员 | 差价（毛利率 30-40%） |
| 第三方+平台订单 | 活动公司 → 平台 → 7天后 → 供应商 | 佣金 12% |
| 第三方+自有客户 | 活动公司 → 供应商（平台不托管） | 0%（订阅已覆盖） |

### 11.6 智能报价引擎全面开放

**报价逻辑：**

```
活动公司询价 → AI分析：
  ① 客户类型（首次/老客户）
  ② 活动类型（年会预算通常高于团建）
  ③ 时间紧迫度（紧急需求可溢价）
  ④ 地理位置（一线城市可溢价）
  ⑤ 历史成交价（同类案例的成交价分布）
  ⑥ 供应商类型（自营/第三方/独立演员）
  → 输出推荐报价区间
```

**价格区间说明：**

| 价格点 | 含义 | 示例（T3级 60分钟 2人） |
|:------:|------|:----------------------:|
| 底线价 | 低于此价不接 | ¥4,800 |
| 推荐价 | 基于历史数据的最佳成交价 | ¥6,000 |
| 目标价 | 可争取的较高价 | ¥7,500 |
| 上限价 | 不应超过的合理价格 | ¥9,000 |

## 12. 技术架构与API规范

### 12.1 现有技术栈

| 层 | 技术 | 版本 |
|:--:|:----|:----:|
| 小程序前端 | uni-app + Vue 3 + Vant Weapp | 最新 |
| PC Web端 | Vite + React 18 + Ant Design 6 | 最新 |
| 运营后台 | Umi 4 + @umijs/max + ProTable | 4.6.69 |
| API | Fastify + TypeScript + Zod | 4.x |
| 数据库 | PostgreSQL | 15 |
| 部署 | 腾讯云香港 2C2G Ubuntu 22.04 | — |

### 12.2 API 设计原则

1. **RESTful：** 所有端点以 `/v1/` 开头
2. **Zod 校验：** 所有请求参数使用 Zod Schema 校验
3. **分页标准：** `page: 1-*, pageSize: 1-1000, 返回 paginatedResponse`
4. **错误码统一：** `0` 成功 / `1xxx` 认证 / `99xx` 参数 / `9999` 服务器错误
5. **JWT 认证：** 所有需要认证的端点使用 `authMiddleware`
6. **API Contract：** 所有 API 变更必须同步更新 `docs/API_CONTRACT.md`

### 12.3 AI 服务接口（Phase 2+）

```
POST /v1/ai/parse-demand
  输入：{ text: string }
  输出：{ demand: { type, activity_type, crowd_size, budget_range, date, location, requirements } }

POST /v1/ai/recommend-plan
  输入：{ demand_id: string }
  输出：{ plans: [{ sku_id, tier, duration, performer_count, price, confidence, reasoning }] }

POST /v1/ai/recommend-price
  输入：{ sku_id, tier, duration, crowd_size, activity_type, location, urgency }
  输出：{ base_price, recommended_price, target_price, ceiling_price, reasoning }
```
