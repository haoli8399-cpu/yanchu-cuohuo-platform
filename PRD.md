## 1. 产品战略定位

### 1.1 竞争策略

**喜剧工厂是「AI驱动的商演交易平台」**，不是纯撮合平台，也不是纯AI工具。

| 维度 | 撮合平台 | AI经纪人工具 | 喜剧工厂（融合模式） |
|:----:|:--------:|:-----------:|:-------------------:|
| 供给端 | 需要地推 | 不需要 | ✅ 已有演员/SKU库 |
| 需求端 | 需要获客 | 获客成本高 | ✅ 已有活动公司 |
| 交易效率 | 人工匹配 | AI自动 | ✅ AI辅助+人工确认 |
| 壁垒 | 网络效应 | 数据+AI | ✅ 网络+AI双重壁垒 |
| 规模化 | 重运营 | 轻模式 | ✅ 逐步降低运营成本 |

### 1.2 核心差异化

> **不是帮用户「管理演出」，而是帮用户「把商演需求变成成交订单」。**

- 竞争对手（活动行、大麦）：演出信息平台，交易在站外
- 我们：商演交易平台，从询价到签约到收款全闭环
- 终局：AI经纪人接管80%标准化工作，人做20%复杂决策

## 2. 产品一句话定义

> **AI驱动的B2B商演交易平台——帮助活动公司15分钟完成从「有个需求」到「拿到报价方案」的全过程，帮助演员/厂牌从零散微信询价走向标准化、可跟进的订单管理。**

## 3. 产品不是什么

| 不是 | 为什么 |
|:----|:----|
| ❌ **不是纯招投标平台** | 活动公司不应该"几家公司竞价"，而是平台基于需求标准化匹配最优方案 |
| ❌ **不是大众点评/大麦网** | 核心用户是B端活动公司，不是C端消费者 |
| ❌ **不是艺人管理系统** | 演员管理是底层数据，不是产品主入口 |
| ❌ **不是全自动AI** | 首版坚持：AI建议+人确认，不做AI自动对外发送 |
| ❌ **不是全行业平台** | 只聚焦脱口秀/喜剧/小剧场商演，不做演唱会/明星经纪 |

## 4. 目标用户与核心场景

### 4.1 用户角色

| 角色 | 定义 | 核心需求 | 使用端 |
|:----:|------|---------|:------:|
| 🏢 **活动公司** | 企业活动策划方、甲方品牌部、年会/团建负责人 | 快速找到合适的演出方案、拿到报价、确认档期 | 小程序 + PC Web端 |
| 🎭 **演员/厂牌** | 脱口秀演员、即兴团队、喜剧厂牌 | 接收排期、确认档期、签到、结算、管理信誉 | 小程序（演员端） |
| 🎯 **运营人员** | 平台运营/经纪人 | 审核需求、匹配演员、调整方案、推进交易 | 运营后台（Web） |
| 👑 **管理员** | 平台管理者 | 数据看板、SKU管理、人员审批 | 运营后台（Web） |

### 4.2 核心用户场景

**场景一：活动公司年会找脱口秀**

```
HR打开小程序 → 
  方式A: 浏览SKU列表，点「脱口秀标准版」，看详情和默认价格，点「获取报价」
  方式B: 输入/粘贴需求文字"年会10月20日，300人，要脱口秀，预算1万以内"
        → AI推荐方案+估价 → 确认提交

提交后 → 运营工作台收到任务 → AI预生成方案+推荐演员 
       → 运营确认/调整 → 点击发送 → 活动公司微信收到方案通知
       → 活动公司确认 → 签约 → 付款 → 演出现场签到 → 评价
```

**场景二：演员管理排期**

```
演员打开小程序 → 看到今日排期 → 看到待确认任务 → 
确认演出 → 当天签到打卡 → 演出后进入结算流程 → 团队收到款 → 信誉分更新
```

## 6. 核心业务流程

### 6.1 活动公司端（完整链路）

```
发现方案 → 查看详情 → 提交需求 → 收到报价 → 确认方案 → 
签约 → 等待演出 → 演出执行 → 评价
```

| 步骤 | 页面 | 功能 |
|:----:|:----:|------|
| 1 | 发现页 `discover` | 搜索/Banner/分类/推荐 |
| 2 | SKU详情 `sku/detail` | 默认配置+价格+介绍+场地+案例+评价 |
| 3 | 提交需求 `request/submit` | 自定义配置器（级别/时长/人数）+ 实时报价 |
| 4 | 需求列表 `request/list` | 状态跟踪 |
| 5 | 需求详情 `request/detail` | AI方案+确认/不采纳+评价 |
| 6 | 个人中心 `user` | 收藏/订单/企业信息/设置/通知 |

### 6.2 演员端

```
收到排期 → 确认档期 → 现场签到 → 结算 → 信誉更新
```

### 6.3 运营端

```
新需求通知 → 查看需求 → AI辅助生成方案 → 调整/确认 → 发送
→ 跟踪确认 → 安排演员 → 跟进执行 → 完成归档
```

## 8. Phase 1 — 运营后台改造（AI辅助运营）

### 8.1 目标

> **运营人员打开后台，看到的不是空菜单，而是「待处理任务」队列。实现「人追着任务走，不是任务等人找」。**

### 8.2 详细功能清单

#### 8.2.1 运营工作台改版

**当前：**
- 运营工作台是一个空白/信息页
- 运营需要逐个点进需求列表去看

**改为：**
```
┌─ 运营工作台 ───────────────────────────────────┐
│  📊 数据总览                                      │
│  ┌─────┐ ┌─────┐ ┌─────┐ ┌─────┐              │
│  │  3  │ │  12 │ │  5  │ │ 87% │              │
│  │待处理│ │今日新│ │跟进中│ │转化率│              │
│  └─────┘ └─────┘ └─────┘ └─────┘              │
│                                                  │
│  🎯 待处理任务（按优先级排列）                     │
│  ┌──────────────────────────────────────────┐   │
│  │ 🔴 高优                                                  │   │
│  │ [需求] XX科技公司年会 — 脱口秀 60分钟     │   │
│  │ 提交时间：7月6日 14:30  已过2小时         │   │
│  │ 🤖 AI推荐方案：脱口秀标准版 T3级  ¥6,000  │   │
│  │ 👤 AI推荐演员：张老师 T3 / 李老师 T4      │   │
│  │ [采用方案] [调整] [更多选项]              │   │
│  └──────────────────────────────────────────┘   │
│  ┌──────────────────────────────────────────┐   │
│  │ 🟡 中优                                                  │   │
│  │ [需求] 某地产项目开盘 — 魔术喜剧 45分钟   │   │
│  │ 提交时间：7月6日 10:00  已过6小时         │   │
│  │ 🤖 AI推荐方案：魔术喜剧标准版 T4级 ¥4,500│   │
│  │ [采用方案] [调整]                        │   │
│  └──────────────────────────────────────────┘   │
└──────────────────────────────────────────────────┘
```

**优先级规则：**
- 🔴 高优：超时未处理（>2小时未响应）
- 🟡 中优：待处理但未超时
- 🟢 低优：已报价待确认、可稍后跟进

#### 8.2.2 AI 方案预生成

**触发时机：** 新需求到达时，系统自动调用 AI 生成推荐方案。

**AI 输入：** 需求描述 + SKU库 + 演员库 + 历史案例

**AI 输出：**

| 字段 | 说明 | 示例 |
|:----|:----|:----|
| 推荐SKU | 匹配度最高的方案模板 | 脱口秀标准版 |
| 推荐级别 | 最佳演员级别 | T3级 |
| 推荐人数 | 建议演员人数 | 2人 |
| 推荐时长 | 建议演出时长 | 60分钟 |
| 推荐演员 | 从演员库中匹配的前3名 | 张老师/T3, 李老师/T4, 王老师/T5 |
| 预估价格 | 甲方标准价 | ¥6,000 |
| 方案说明 | AI生成的一段文字 | "该方案适合300人年会，60分钟包含..." |
| 历史参考 | 同类案例链接 | XX科技公司年会 2024 |

#### 8.2.3 方案调整工具

运营可以对 AI 生成的方案进行调整：

- **调整SKU：** 下拉选择其他方案
- **调整演员级别：** T0-T6 滑块
- **调整时长：** 15分钟为单位增减
- **调整人数：** 最少/最多人数
- **替换演员：** 从演员库选择其他演员
- **修改价格：** 手动输入（自动计算渠道价）
- **添加备注：** 运营备注（不对外）

**确认后：** 点击「发送方案」，活动公司在需求详情页看到方案卡片。

## 9. Phase 2 — 需求入口升级（自然语言需求）

### 9.1 目标

> **让活动公司不需要「理解SKU体系」就能提交需求。输入/粘贴一段需求文字，AI 自动匹配方案和估价。**

### 9.2 功能设计

#### 9.2.1 入口位置

AI 聊天窗口放在**小程序「提需求」Tab 内**（不是独立页面，不做全局悬浮按钮）。

**为什么是这里：**
- ✅ 小程序不支持全局悬浮按钮（wx api限制）
- ✅ 用户点"提需求"就是想描述需求——**点提需求就是点开AI经纪人**
- ✅ 不需要用户"找到AI入口"——入口就是Tab本身
- ❌ 独立页面：用户需要离开当前路径去找AI，多一次跳转就多一层流失

**页面结构：**

```
小程序底部Tab：发现 | 方案 | 提需求 | 消息 | 我的
                          ↑
                   点此直接打开AI经纪人对话窗口
```

#### 9.2.2 对话式交互设计

「提需求」页面改为双Tab结构：

> **Tab 1「📋 选方案提交」** — 保留现有 SKU 浏览+提交流程

#### 9.2.3 AI 主动追问机制

**必须追问，否则就不是AI经纪人。** 真实场景中活动公司HR通常只写一句模糊需求（"年会想搞个脱口秀"），不追问无法匹配方案。

**追问策略：**

```
用户: "年会想搞个脱口秀"
         ↓
AI: 好的！我先确认几个信息：
    ① 多少人参加？
      [50-100人] [100-300人] [300-500人] [500人以上]

用户: 选 100-300人
         ↓
AI: ② 预算范围？
      [5千以内] [5千-1万] [1万-2万] [2万以上] [还没想好]

用户: 1-2万
         ↓
AI: ③ 大概什么时间？
      [请选择日期]

用户: 选 10月20日
         ↓
AI: 明白了——300人年会，10月20日，预算1-2万。
    推荐「脱口秀标准版」T3级 60min ¥6,000 →
    再加一个备选「脱口秀旗舰版」T2级 90min ¥9,000 →
    [确认提交]
```

**追问规则：**

| 缺失字段 | 追问方式 | 是否必填 |
|:--------|:--------|:--------:|
| 人数 | 提供几个常见范围按钮（快速选择） | 是（影响演员配置） |
| 日期 | 日期选择器 | 是（影响档期匹配） |
| 预算 | 几个价格区间按钮 | 是（影响SKU推荐） |
| 地点 | 输入框 | 否（可选，辅助场地匹配） |
| 特殊要求 | 输入框 | 否（所有其他要求写在这里） |

**追问次数上限：最多3轮。3轮后使用已有信息直接匹配。** 避免把用户困在问卷里。

**追问状态指示：**

```
追问中（如"①/③ 还需补充2个信息"）
追问完成 → 自动开始匹配
匹配中（如"🤖 正在为你匹配方案..."）→ 展示结果卡片
```

#### 9.2.4 AI 自然语言识别

**输入：** 自由文本（上限500字）

**识别字段：**

| 字段 | 识别方式 | 示例 |
|:----|:--------|:----|
| 演出类型 | 关键词匹配 + 语义理解 | "脱口秀""即兴""魔术""喜剧" |
| 活动类型 | 上下文判断 | "年会""团建""发布会""开盘" |
| 人数规模 | 数字提取 | 300人 |
| 预算范围 | 数字+语义 | "1万以内""5千-8千" |
| 日期/时段 | 日期识别 | 10月20日/下个月初 |
| 地点 | 地名识别 | 上海浦东 |
| 特殊要求 | 语义理解 | "要女演员""最好会互动""不要太黄段子" |

#### 9.2.3 推荐匹配逻辑

```
输入文字 → AI 结构化 → 
  Step 1: 匹配演出类型 → 过滤SKU库
  Step 2: 匹配预算 → 过滤价格区间
  Step 3: 匹配人数 → 推荐演员人数
  Step 4: 匹配日期 → 检查档期
  Step 5: 历史参考 → 查询同类案例成交价
  → 输出 Top 2-3 方案
```

## 10. Phase 3 — AI经纪人平台（交易闭环）

### 10.1 目标

> **从撮合平台升级为 AI 经纪人平台：AI 处理80%的标准化流程，人处理20%的复杂决策。引入订阅制商业模式。**

### 10.2 核心功能

#### 10.2.1 AI 经纪人订阅制

| 套餐 | 价格 | 包含功能 |
|:----:|:----:|:--------:|
| **免费版** | ¥0 | 浏览SKU、手动提交需求、等待运营匹配 |
| **标准版** | ¥299/月 | **AI自动报价**、智能方案推荐、3次免费跟进提醒 |
| **专业版** | ¥599/月 | AI自动报价+方案、**优先运营响应**、**在线签约**、无限跟进提醒、历史报价对比 |

**计费规则：**
- 免费试用期 14 天
- 按月订阅，自动续费
- 非成交可取消 → 选择放弃跟进
- 成交不额外收费（不抽佣金）

#### 10.2.2 智能跟进提醒

**触发规则：**

| 场景 | 提醒间隔 | 提醒内容 |
|:----|:--------|:--------|
| 方案已发送，未确认 | 24小时 / 48小时 / 72小时 | 「XX方案已发送，请确认」 |
| 方案已确认，未签约 | 24小时 / 48小时 | 「方案已确认，请完成签约」 |
| 签约后未支付 | 到期前 3 天 / 1 天 | 「演出倒计时，请完成支付」 |
| 演出前一天 | 自动提醒 | 「明天演出，请准备好」 |

**提醒渠道：** 小程序通知 + 短信（可选）

#### 10.2.3 在线签约

**流程：**
```
方案确认 → AI 生成合同草稿 → 
  运营 审阅/编辑 → 
  活动公司 查看/签署 → 
  演员/厂牌 签署 → 
  合同归档
```

**合同模板字段：**
- 演出方/需求方信息
- 演出时间/时长/地点
- 演员阵容/级别
- 价格/付款方式（定金50%+尾款50%）
- 取消条款
- 设备要求
- 知识产权条款

#### 10.2.4 支付闭环

**支付流程：**

```
方案确认 → 合同签署 → 支付定金（50%）→ 
演员安排 → 演出执行 → 活动公司确认完成 → 
支付尾款（50%）→ 演员结算
```

**资金流向：**
- 活动公司 → 平台托管 → 演出后7天 → 演员/厂牌
- 平台保留0%佣金（订阅制已收费），仅收取支付通道费（微信支付0.6%）

#### 10.2.5 智能报价系统

**报价逻辑（Phase 3 升级）：**

```
活动公司询价 → AI分析：
  ① 客户类型（首次/老客户）
  ② 活动类型（年会预算通常高于团建）
  ③ 时间紧迫度（紧急需求可溢价）
  ④ 地理位置（一线城市可溢价）
  ⑤ 历史成交价（同类案例的成交价分布）
  → 输出推荐报价区间
```

**价格区间说明：**

| 价格点 | 含义 | 示例（T3级 60分钟 2人） |
|:------:|------|:----------------------:|
| 底线价 | 低于此价不接 | ¥4,800 |
| 推荐价 | 基于历史数据的最佳成交价 | ¥6,000 |
| 目标价 | 可争取的较高价 | ¥7,500 |
| 上限价 | 不应超过的合理价格 | ¥9,000 |

## 11. 技术架构与API规范

### 11.1 现有技术栈

| 层 | 技术 | 版本 |
|:--:|:----|:----:|
| 小程序前端 | uni-app + Vue 3 + Vant Weapp | 最新 |
| PC Web端 | Vite + React 18 + Ant Design 6 | 最新 |
| 运营后台 | Umi 4 + @umijs/max + ProTable | 4.6.69 |
| API | Fastify + TypeScript + Zod | 4.x |
| 数据库 | PostgreSQL | 15 |
| 部署 | 腾讯云香港 2C2G Ubuntu 22.04 | — |

### 11.2 API 设计原则

1. **RESTful：** 所有端点以 `/v1/` 开头
2. **Zod 校验：** 所有请求参数使用 Zod Schema 校验
3. **分页标准：** `page: 1-*, pageSize: 1-1000, 返回 paginatedResponse`
4. **错误码统一：** `0` 成功 / `1xxx` 认证 / `99xx` 参数 / `9999` 服务器错误
5. **JWT 认证：** 所有需要认证的端点使用 `authMiddleware`
6. **API Contract：** 所有 API 变更必须同步更新 `docs/API_CONTRACT.md`

### 11.3 AI 服务接口（Phase 2+）

```
POST /v1/ai/parse-demand
  输入：{ text: string }
  输出：{ demand: { type, activity_type, crowd_size, budget_range, date, location, requirements } }

POST /v1/ai/recommend-plan
  输入：{ demand_id: string }
  输出：{ plans: [{ sku_id, tier, duration, performer_count, price, confidence, reasoning }] }

POST /v1/ai/recommend-price
  输入：{ sku_id, tier, duration, crowd_size, activity_type, location, urgency }
  输出：{ base_price, recommended_price, target_price, ceiling_price, reasoning }
```
