## 1. 产品战略定位

### 1.1 当前阶段定位（V3.3.2）

> Phase 1 当前不优先建设开放平台，而是优先建设 AI 辅助商演成交机器。

Phase 1 核心目标：让自营运营完成 模糊需求识别→商机判断→快速报价→持续跟进→成交/丢单复盘。
长期定位：喜剧工厂最终演进为 AI 驱动的 B2B 商演交易平台。

### 1.2 竞争策略

**喜剧工厂是「AI驱动的商演交易平台」**，不是纯撮合平台，也不是纯AI工具。

| 维度 | 撮合平台 | AI经纪人工具 | 喜剧工厂（融合模式） |
|:----:|:--------:|:-----------:|:-------------------:|
| 供给端 | 需要地推 | 不需要 | ✅ 已有艺人/SKU库 |
| 需求端 | 需要获客 | 获客成本高 | ✅ 已有活动公司 |
| 交易效率 | 人工匹配 | AI自动 | ✅ AI辅助+人工确认 |
| 壁垒 | 网络效应 | 数据+AI | ✅ 网络+AI双重壁垒 |
| 规模化 | 重运营 | 轻模式 | ✅ 逐步降低运营成本 |

### 1.3 核心差异化

> **不是帮用户「管理演出」，而是帮用户「把商演需求变成成交订单」。**

- 竞争对手（活动行、大麦）：演出信息平台，交易在站外
- 我们：商演交易平台，从询价到签约到收款全闭环
- 终局：AI经纪人接管80%标准化工作，人做20%复杂决策

## 2. 产品一句话定义

> **AI驱动的B2B商演交易平台——帮助活动公司15分钟完成从「有个需求」到「拿到报价方案」的全过程，帮助艺人/厂牌从零散微信询价走向标准化、可跟进的订单管理。**

## 3. 产品不是什么

| 不是 | 为什么 |
|:----|:----|
| ❌ **不是纯招投标平台** | 活动公司不应该"几家公司竞价"，而是平台基于需求标准化匹配最优方案 |
| ❌ **不是大众点评/大麦网** | 核心用户是B端活动公司，不是C端消费者 |
| ❌ **不是艺人管理系统** | 艺人管理是底层数据，不是产品主入口 |
| ❌ **不是全自动AI** | 首版坚持：AI建议+人确认，不做AI自动对外发送 |
| ❌ **不是全行业平台** | 只聚焦脱口秀/喜剧/小剧场商演，不做演唱会/明星经纪 |

## 4. 目标用户与核心场景

### 4.1 用户角色

最终角色清单（7类）：

| 角色 | 归属于 | 使用端 | 核心功能 | 阶段 |
|:----|:------|:------|:--------|:----:|
| 🏢 活动公司 | 演艺电商 | 小程序 + PC Web | 浏览方案、比价、提交需求、查看渠道价 | Phase 1 |
| 🏢 直接甲方 | 演艺电商 | 小程序 + PC Web | 同上，但看到的是甲方标准价 | Phase 1 |
| 🔧 平台管理员 | 演艺电商 | platform-admin (PC) | 供应商管理、佣金设置、全平台看板、仲裁 | Phase 1 |
| 🎯 自营运营 | 演艺电商 | supplier-console (PC) | 接自营订单、管艺人、SKU管理、看利润 | Phase 1 |
| 🏪 第三方经纪公司 | 演艺电商(平台订单) + Agent(自有客户) | supplier-console (PC) | 平台接单 + Agent处理自有客户 | Phase 3 |
| 🎤 独立艺人 | 演艺电商(平台订单) + Agent(自有客户) | supplier-console小程序版 + 艺人模式 | 接单、管SKU、看收入 | Phase 3 |
| 🎭 经纪公司旗下艺人 | AI商演经纪人（不含电商平台） | 小程序艺人模式 | 排期、签到、结算、信誉 | Phase 3 |

> Phase 1 只实现前4类角色，其余为Phase 3开放平台预留。

**角色关系说明：**

- **活动公司与直接甲方是两个不同角色**，核心区别：活动公司看到渠道价，直接甲方看到甲方标准价
- **经纪公司旗下艺人不直接出现在演艺电商平台**——他们由经纪公司在 Agent 内管理，仅通过小程序艺人模式查看排期、签到、结算
- **独立艺人是特殊角色**：既是 supplier（管理自己SKU、接单）又是 performer（排期、演出、签到、结算）。两个模式都能用
- **自营运营、第三方经纪公司共用同一套 supplier-console**（PC为主，小程序做核心操作补充）
- **平台管理员使用独立的 platform-admin**
- 小程序提供两个角色入口：
  - **活动公司/直接甲方模式**：默认 tab 列（首页 | 找方案 | 提需求 | 消息 | 我的）
  - **艺人模式**：切换后显示排期/结算/签到 tab

### 4.2 核心用户场景

**场景一：活动公司年会找脱口秀（已有）**

```
HR打开小程序 → 
  方式A: 浏览SKU列表，点「脱口秀标准版」，看详情和渠道价格，点「获取报价」
  方式B: 输入/粘贴需求文字"年会10月20日，300人，要脱口秀，预算1万以内"
        → AI推荐方案+估价 → 确认提交

提交后 → 运营工作台收到任务 → AI预生成方案+推荐艺人 
       → 运营确认/调整 → 点击发送 → 活动公司微信收到方案通知
       → 活动公司确认 → 签约 → 付款 → 演出现场签到 → 评价
```

> 注：直接甲方的流程同上，区别在于看到的SKU价格为甲方标准价而非渠道价。

**场景二：艺人管理排期（已有）**

```
艺人打开小程序 → 看到今日排期 → 看到待确认任务 → 
确认演出 → 当天签到打卡 → 演出后进入结算流程 → 团队收到款 → 信誉分更新
```

**场景三：自营运营处理需求（新增）**

```
自营运营打开 supplier-console (PC) → 
工作台展示新需求卡片 + AI预生成方案 →
  查看AI推荐的SKU+艺人+报价 →
  调整（换艺人级别/改时长/调价格） →
  点击「发送方案」→ 活动公司收到通知 →
  跟踪确认状态 → 活动公司确认后安排艺人 → 跟进演出执行
```

**场景四：第三方经纪公司接单（新增）**

```
第三方经纪公司打开 supplier-console (PC) → 
工作台收到平台分配的需求 →
  查看需求详情（人数/预算/日期/活动类型） →
  基于平台推荐报价，自主报价（可调整） →
  安排旗下艺人（选人+定档期） →
  提交方案 → 活动公司端收到方案 →
  确认后签约 → 演出执行 → 结算 → 平台抽佣
```

**场景五：独立艺人管理自己（新增）**

```
独立艺人打开小程序，切换到「艺人模式」→ 
  查看排期日历 + 待确认演出 →
  
同时打开 supplier-console 小程序版 →
  管理个人SKU（价格/时长/类型/可演出日期） →
  查看历史订单和收入 →
  收到平台订单 → 确认/拒绝 →
  演出 → 签到 → 结算 → 收入到账
```

### 4.3 小程序 vs PC Web 的差异化定位

| 场景 | 小程序（主力） | PC Web（补充） |
|:----|:-----------|:------------|
| 快速浏览 | ✅ 手势操作、微信登录 | — |
| 提交需求 | ✅ AI对话+快速表单 | ⚠️ 复杂需求时可用 |
| 深度方案比价 | — | ✅ 多列对比、筛选器 |
| 查看/跟进订单 | ✅ 消息推送 | ⚠️ 查看 |
| 审阅/签署合同 | — | ✅ 大屏阅读、下载打印 |
| 数据复盘 | — | ✅ 历史成交数据查看 |

**策略：** 小程序是「快速决策+看状态」的移动入口，PC Web 是「深度对比+合同管理」的工作站。两个端功能集有差异，不追求一一对应。

## 6. 核心业务流程

### 6.1 活动公司端（完整链路）

**路径A：AI快速提需求**
首页输入/粘贴需求 → AI识别 → 补全字段 → 提交需求 → 收到报价 → 确认方案

**路径B：浏览标准方案**
找方案 → SKU详情 → 获取报价 → 提交需求 → 收到报价 → 确认方案

> 注：直接甲方的流程同上，唯一区别是SKU中展示的价格为甲方标准价。

### 6.2 艺人端

```
收到排期 → 确认档期 → 现场签到 → 结算 → 信誉更新
```

### 6.3 运营端

```
新需求通知 → 查看需求 → AI辅助生成方案 → 调整/确认 → 发送
→ 跟踪确认 → 安排艺人 → 跟进执行 → 完成归档
```

### 6.4 商机状态机

> 核心原则：每个需求都是一条商机，从进入到关闭必须经过明确的状态流转。运营人员根据状态执行对应动作，系统根据状态自动触发提醒。

**商机状态流转：**

```
                    ┌─────────────┐
                    │    new      │  ← 新需求到达
                    │   新需求     │
                    └──────┬──────┘
                           │ AI解析+人工审核
                           ▼
                    ┌─────────────┐
              ┌─────│ qualified   │  ← 需求有效，进入报价
              │     │   有效      │
              │     └──────┬──────┘
              │            │ 运营生成报价
              │            ▼
              │     ┌─────────────┐
              │     │   quoted    │  ← 报价已发送
              │     │  已报价     │
              │     └──────┬──────┘
              │            │ 客户有异议/还价
              │            ▼
              │     ┌─────────────┐
              │     │ negotiating │  ← 谈判中
              │     │  谈判中     │
              │     └──┬──────┬──┘
              │        │      │ 等待客户决策
              │        │      ▼
              │        │ ┌──────────────┐
              │        │ │pending_client│ ← 等客户
              │        │ │  等客户      │
              │        │ └──┬───────┬──┘
              │        │    │       │
              │        ▼    ▼       ▼
              │     ┌──────────┐ ┌──────────┐
              │     │   won    │ │   lost   │
              │     │  成交     │ │  丢单    │
              │     └──────────┘ └──────────┘
              │
              └──  new → invalid（无效需求直接关闭）
```

**各状态对应运营动作：**

| 状态 | 运营动作 | 自动提醒 |
|:-----|:--------|:--------|
| **invalid** | 无效需求（测试、非商演需求、重复提交），立即关闭，不进入后续流程 | 自动归档，无提醒 |
| **new** | AI解析需求、人工确认信息完整性 | 新需求到达即时通知 |
| **qualified** | 确认需求有效、选择SKU模板、匹配艺人 | 2h内未处理→升级提醒 |
| **quoted** | 报价已发送、等待客户查看 | 24h未读→提醒运营；48h未回复→AI生成跟进话术 |
| **negotiating** | 客户有异议、调整方案/价格 | 每次沟通后24h无进展→提醒 |
| **pending_client** | 等待客户最终决策 | 48h无动静→AI生成温和提醒话术 |
| **won** | 进入签约+排期+演出执行流程 | 自动归档 |
| **lost** | 记录丢单原因、分析复盘 | 周报汇总丢单分析 |

**丢单原因（必须记录）：**

| 原因分类 | 说明 |
|:--------|:----|
| 预算太低 | 客户预算无法覆盖最低方案 |
| 日期不合适 | 所有艺人档期已满 |
| 客户无回应 | 报价后客户失联 |
| 被竞品 | 客户选择了其他供应商 |
| 方案不匹配 | 没有合适类型的方案 |
| 客户取消活动 | 活动本身取消 |
| 价格太高 | 客户认为价格超出预期 |
| 非真实需求 | 询价但实际无购买意向 |

> **设计原则：** 丢单和成交同样重要——不知道为什么丢单，就无法优化成交效率。Phase 1 开始就必须记录丢单原因，不可省略。

### 6.5 报价版本管理

> 报价不是一次定稿的。客户嫌贵就降配、追加需求就升级方案——每次调整都应记录为一个版本，可追溯、可分析。

**报价版本示例：**

| 版本 | 方案 | 价格 | 状态 | 备注 |
|:----:|------|:----:|:----:|:----:|------|
| v1 | T3 脱口秀 60min 2人 | ¥6,000 | 已发送 | 初始报价 |
| v2 | T4 脱口秀 45min 2人 | ¥4,500 | 已发送 | 客户嫌贵，降配 |
| v3 | T3 脱口秀+互动 90min 3人 | ¥8,000 | 已确认 | 客户加互动需求 |

**报价版本数据模型（quotes 表）：**

| 字段 | 类型 | 说明 |
|:-----|:----|:----|
| `id` | UUID | 报价单ID |
| `demand_id` | UUID | 关联需求 |
| `opportunity_id` | UUID | 关联商机 |
| `version` | int | 版本号（v1, v2, v3...） |
| `total_price` | decimal | 总价（甲方标准价） |
| `channel_price` | decimal | 活动公司渠道价 |
| `cost_price` | decimal | 艺人成本（内部） |
| `margin` | decimal | 毛利（total - cost） |
| `valid_until` | timestamp | 报价有效期 |
| `status` | enum | draft/sent/viewed/confirmed/rejected/expired |
| `rejection_reason` | text | 客户拒绝原因 |
| `ai_generated` | boolean | 是否AI生成 |
| `operator_modified` | boolean | 运营是否手动修改过 |
| `created_at` | timestamp | 创建时间 |

**版本管理规则：**
- 同一需求可以有多个报价版本
- 客户每次反馈（嫌贵/改需求）→ 创建新版本
- 旧版本保留历史记录，不可删除
- 最终确认的版本标记为 `confirmed`
- 报价对比：可查看 v1→v2→v3 的变化轨迹（做了什么调整、为什么）

### 6.6 争议处理流程（Phase 2+ 系统化）

> **Phase 1：** 采用人工 SOP + 记录（在 follow_up_logs 和 dispute_records 中记录）。  
> **Phase 2+：** 推进系统化争议裁决流程如下。

演出服务具有不可逆性，线下问题（质量、迟到、换人）无法自动判定，需建立标准化争议裁决机制。

**争议类型与赔付标准：**

| 争议类型 | 判定方式 | 赔付方案 |
|:--------|:--------|:--------|
| 艺人迟到 | 签到时间对比演出时间 | 迟到30分钟以上→减免20%费用 |
| 演出缩水 | 现场协调人确认 | 按比例退款 |
| 临时换人 | 未获甲方同意 | 减免30%费用 |
| 演出质量严重不符 | 甲方举证+平台仲裁 | 退款50-100% |
| 甲方无故取消 | 签约时间判断 | 24h内取消→定金不退；24h+→退还50% |

**争议 SOP：** 任一方发起→平台运营48h内仲裁→资金按裁决解冻→记录入 dispute_records 表→影响供应商信誉分。

## 7. 商业模式

### 7.1 模型定位：京东自营模型

> 喜剧工厂借鉴京东自营模式——平台自己是市场管理者，"喜剧工厂自营"是平台上一家供应商。活动公司在方案列表中看到的是「混排」结果：自营方案和第三方供应商方案并列展示，通过供应商标识区分。

### 7.2 三层渐进收入模型

#### Tier 1：自营赚差价（当前阶段）

- **模式：** 喜剧工厂签约艺人，作为唯一供应商
- **盈利方式：** 赚取演出差价（活动公司付¥6,000 → 艺人出场费¥4,000 → 平台利润¥2,000）
- **目标毛利率：** 30-40%
- **启动条件：** 现在（已有艺人库和SKU体系）
- **关键动作：** 定价策略 + 运营效率提升

#### Tier 2：平台抽佣（月需求≥100单后开放）

- **模式：** 第三方经纪公司/独立艺人入驻，与自营混排展示
- **盈利方式：**
  - 平台带来的订单 → 抽佣 **10-15%**（按订单金额计算）
  - 第三方自己带来的订单 → **不抽佣**
- **目标佣金率：** 12%
- **启动条件：** 月需求≥100单（约 Phase 2 中后期）
- **关键动作：** 供应商入驻流程 + 佣金结算系统 + 品质管控

#### Tier 3：AI Agent 订阅（Agent 打磨成熟后开放）

- **模式：** 第三方经纪公司/独立艺人可订阅 AI 商演经纪人处理商演邀约
- **定价：**
  - 个人版：**¥199/月**（独立艺人用）
  - 企业版：**¥499/月**（经纪公司用）
- **弹性定价：** 年付 9 折（¥179 / ¥449），首月免费试用。月询价量 > 50 单可定制企业方案。
- **核心功能：** 商机判断、AI 报价、回复话术生成、智能跟进提醒、合同生成、谈判建议
- **规则：** 订阅用户的自有渠道订单 → 0 佣金（订阅费已覆盖）
- **启动条件：** AI Agent 功能成熟、供应商量达到 10+ 家
- **关键动作：** Agent 产品化 + 订阅计费系统 + 非平台来源订单管理

### 7.3 三层收入关系

三层模型不是替代关系，而是叠加关系：

| 收入来源 | Tier 1 | Tier 2 | Tier 3 |
|:--------|:-----:|:-----:|:-----:|
| 自营差价 | ✅ | ✅ | ✅ |
| 平台佣金 | — | ✅ | ✅ |
| AI Agent 订阅 | — | — | ✅ |

**逐步演进示意：**

```
Tier 1 阶段（当前）
  收入 = 自营差价
  月营收目标：¥3.6万（假设月订单10-15单，均价¥6,000，毛利40%）

Tier 1+2 阶段（Phase 2 中后期）
  收入 = 自营差价 + 平台佣金
  自营：保持不变
  第三方：假设月订单20单，均价¥5,000，佣金12%
  月营收目标：自营¥3.6万 + 佣金¥1.8万 = ¥5.4万

Tier 1+2+3 阶段（Phase 3）
  收入 = 自营差价 + 平台佣金 + Agent订阅
  自营：¥3.6万
  第三方佣金：¥3万（月订单50单，均价¥5,000，佣金12%）
  Agent订阅：¥7,500（5家公司×¥499 + 25个艺人×¥199）
  月营收目标：¥7.35万
```

### 7.4 价格体系

> **核心原则：同一场演出，不同角色看到不同价格。价格不是产品属性，是角色权限。**

**三层价格定义：**

| 价格类型 | 谁看到 | 含义 | 计算方式 |
|:--------|:-----|:----|:------|
| **甲方标准价** | 未登录用户 + 直接甲方 | 面向终端客户的公开统一报价 | SKU中配置 |
| **活动公司渠道价** | 活动公司（登录后） | 分销渠道成本价 = 标准价 × 0.7 | 标准价 × 折扣系数（0.7） |
| **艺人成本** | 仅平台运营 + 管理员（内部） | 平台与艺人/供应商的内部结算价 | 手动输入 |

**价格展示规则：**

- 登录后根据角色显示对应价格
- **活动公司登录** → SKU列表/详情显示渠道价，小字提示"甲方标准价 ¥X,XXX"
- **直接甲方登录** → SKU列表/详情显示甲方标准价
- **提交需求页配置器** → 根据登录角色取对应的价格基础计算（渠道价或标准价）
- **未登录** → 显示甲方标准价（公开价）

**第三方价格：**

- 第三方供应商自行定价，不受平台标准价体系约束
- 第三方SKU展示价格 = 供应商自主设定价格
- 平台在此基础上收取佣金 12%

**已有价格体系补充（兼容不改）：**

| 价格类型 | 说明 | 适用对象 |
|:--------|:----|:--------|
| 甲方标准价 | 对终端甲方展示的公开报价 | 未登录用户/直接甲方/活动公司转发甲方时使用 |
| 活动公司渠道价 | 活动公司的成本价（= 标准价 × 0.7） | 活动公司 |
| 内部结算价 | 平台给艺人的出场费 | 自营艺人 |
| 第三方自主定价 | 第三方供应商自行定价 | 第三方订单 |
| Agent 订阅费 | 个人¥199/月，企业¥499/月 | Tier 3 订阅用户 |

#### 9.2.3 方案调整工具

运营可以对 AI 生成的方案进行调整：

- **调整SKU：** 下拉选择其他方案
- **调整艺人级别：** T0-T6 滑块
- **调整时长：** 15分钟为单位增减
- **调整人数：** 最少/最多人数
- **替换艺人：** 从艺人库选择其他艺人

#### 9.2.4 AI 推荐反馈回路（4类细化反馈）

运营对 AI 的反馈不是简单的「好/不好」，而是按4个维度分别评价。每次反馈都是AI模型的训练数据点。

**反馈分类：**

**① 需求识别反馈**

运营判断 AI 对客户需求的理解是否准确：

```
🤖 AI解析：300人年会，预算1-2万，脱口秀
[✅ 准确]  [❌ 人数错了]  [❌ 日期错了]  [❌ 预算错了]  [💬 补充]
```

反馈维度：人数、日期、预算、活动类型、演出类型、地点

**② 方案推荐反馈**

运营判断 AI 推荐的 SKU 方案是否合适：

```
🤖 AI推荐：脱口秀标准版 T3级 60min ¥6,000
[✅ 合适]  [❌ SKU类型不匹配]  [❌ 时长不合适]  [❌ 人数配置不合适]  [💬 其他]
```

反馈维度：SKU类型、艺人级别、演出时长、艺人人数

**③ 报价反馈**

运营判断 AI 的报价是否合理：

```
🤖 AI报价：甲方标准价 ¥6,000 / 渠道价 ¥4,200 / 毛利 ¥2,000
[✅ 合理]  [📈 价格偏高]  [📉 价格偏低]  [⚠️ 毛利不足(<25%)]  [💬 调整建议]
```

反馈维度：标准价合理性、渠道价合理性、毛利率是否达标

**④ 艺人推荐反馈**

运营判断 AI 推荐的艺人是否合适：

```
🤖 AI推荐艺人：张老师 T3 / 李老师 T4 / 王老师 T5
[✅ 合适]  [📅 档期不符]  [⭐ 咖位不符]  [🎭 风格不符]  [🛡️ 信誉不足]  [💬 其他]
```

反馈维度：档期匹配度、咖位匹配度、风格匹配度、信誉分

**反馈数据流：**

```
运营反馈 → 记录入 ai_feedback_logs 表（携带具体维度和理由）
         → 每周汇总分析：哪些维度反馈最多？
         → 针对性优化模型 Prompt/参数
         → 长期积累 RLHF 训练数据集
```

> **设计原则：** 模糊的「好/不好」无法驱动模型改进。必须把反馈拆解到具体维度，才能定位问题、持续优化。

#### 10.3.3 AI 主动追问机制

**必须追问，否则就不是AI经纪人。** 真实场景中活动公司HR通常只写一句模糊需求（"年会想搞个脱口秀"），不追问无法匹配方案。

**追问策略：**

```
用户: "年会想搞个脱口秀"
         ↓
AI: 好的！我先确认几个信息：
    ① 多少人参加？
      [50-100人] [100-300人] [300-500人] [500人以上]

用户: 选 100-300人
         ↓
AI: ② 预算范围？
      [5千以内] [5千-1万] [1万-2万] [2万以上] [还没想好]

用户: 1-2万
         ↓
AI: ③ 大概什么时间？
      [请选择日期]

用户: 选 10月20日
         ↓
AI: 明白了——300人年会，10月20日，预算1-2万。
    推荐「脱口秀标准版」T3级 60min ¥6,000 →
    再加一个备选「脱口秀旗舰版」T2级 90min ¥9,000 →
    [确认提交]
```

**追问规则：**

| 缺失字段 | 追问方式 | 是否必填 |
|:--------|:--------|:--------:|
| 人数 | 提供几个常见范围按钮（快速选择） | 是（影响艺人配置） |
| 日期 | 日期选择器 | 是（影响档期匹配） |
| 预算 | 几个价格区间按钮 | 是（影响SKU推荐） |
| 地点 | 输入框 | 否（可选，辅助场地匹配） |
| 特殊要求 | 输入框 | 否（所有其他要求写在这里） |

**追问次数上限：最多3轮。3轮后使用已有信息直接匹配。** 避免把用户困在问卷里。

**追问状态指示：**

```
追问中（如"①/③ 还需补充2个信息"）
追问完成 → 自动开始匹配
匹配中（如"🤖 正在为你匹配方案..."）→ 展示结果卡片
```

#### 10.3.4 AI 自然语言识别

**输入：** 自由文本（上限500字）

**识别字段：**

| 字段 | 识别方式 | 示例 |
|:----|:--------|:----|
| 演出类型 | 关键词匹配 + 语义理解 | "脱口秀""即兴""魔术""喜剧" |
| 活动类型 | 上下文判断 | "年会""团建""发布会""开盘" |
| 人数规模 | 数字提取 | 300人 |
| 预算范围 | 数字+语义 | "1万以内""5千-8千" |
| 日期/时段 | 日期识别 | 10月20日/下个月初 |
| 地点 | 地名识别 | 上海浦东 |
| 特殊要求 | 语义理解 | "要女艺人""最好会互动""不要太黄段子" |

#### 10.3.5 推荐匹配逻辑

```
输入文字 → AI 结构化 → 
  Step 1: 匹配演出类型 → 过滤SKU库
  Step 2: 匹配预算 → 过滤价格区间
  Step 3: 匹配人数 → 推荐艺人人数
  Step 4: 匹配日期 → 检查档期
  Step 5: 历史参考 → 查询同类案例成交价
  → 输出 Top 2-3 方案
```

#### 10.3.6 信息提取预览

#### 11.5.1 签约流程

```
方案确认 → AI 生成合同草稿 →
  供应商审阅/编辑 →
  活动公司查看/签署 →
  供应商签署 →
  合同归档 → 进入支付流程

#### 11.5.2 支付流程

```
合同签署 → 活动公司支付定金（50%）→
  艺人安排 → 演出执行 → 活动公司确认完成 →
  支付尾款（50%）→ 供应商结算

#### 11.5.3 资金流向

| 订单类型 | 资金流向 | 平台收费 |
|:--------|:--------|:--------|

| 第三方+自有渠道 | 活动公司 → 供应商（平台不托管） | 0%（订阅已覆盖） |

### 11.6 智能报价引擎全面开放

**报价逻辑：**

```
活动公司询价 → AI分析：
  ① 客户类型（首次/老客户）
  ② 活动类型（年会预算通常高于团建）
  ③ 时间紧迫度（紧急需求可溢价）
  ④ 地理位置（一线城市可溢价）
  ⑤ 历史成交价（同类案例的成交价分布）
  ⑥ 供应商类型（自营/第三方/独立艺人）
  → 输出推荐报价区间
```

**价格区间说明：**

| 价格点 | 含义 | 示例（T3级 60分钟 2人） |
|:------:|------|:----------------------:|
| 底线价 | 低于此价不接 | ¥4,800 |
| 推荐价 | 基于历史数据的最佳成交价 | ¥6,000 |
| 目标价 | 可争取的较高价 | ¥7,500 |
| 上限价 | 不应超过的合理价格 | ¥9,000 |

### 11.7 品牌隔离机制

第三方方案在 SKU 列表和详情页以 `由第三方提供` 小字标注区分，自营方案标注 `🏷️ 自营` 标签。品牌责任分离——活动公司在浏览方案时明确知道与哪方交易，出现争议时可追溯。

### 11.8 新供应商试用期保障

所有新入驻供应商前 5 单享受平台额外保障：
- 活动公司首单不满意 → 全额退款（平台承担赔付）
- 标记 `🛡️ 首单保障` 标签，降低甲方首次合作的决策风险
- 5 单后根据实际评价转为正常供应商

### 11.9 精选邀请制

Phase 1 后期即邀请 3-5 家优质经纪公司以「邀请制」入驻，降低 Phase 2 冷启动门槛：
- 入选条件：已有 10 个以上成功案例、平台审核通过
- 邀请制供应商享有与自营同等的方案展示权
- 提前验证平台多供应商运营流程，发现问题早修正

### 12.1 现有技术栈

### 12.2 API 设计原则

1. **RESTful：** 所有端点以 `/v1/` 开头
2. **Zod 校验：** 所有请求参数使用 Zod Schema 校验
3. **分页标准：** `page: 1-*, pageSize: 1-1000, 返回 paginatedResponse`
4. **错误码统一：** `0` 成功 / `1xxx` 认证 / `99xx` 参数 / `9999` 服务器错误
5. **JWT 认证：** 所有需要认证的端点使用 `authMiddleware`
6. **API Contract：** 所有 API 变更必须同步更新 `docs/API_CONTRACT.md`

### 12.3 AI 服务接口（Phase 2+）

POST /v1/ai/parse-demand

  输出：{ demand: { type, activity_type, crowd_size, budget_range, date, location, requirements } }

POST /v1/ai/recommend-plan
  输入：{ demand_id: string }
  输出：{ plans: [{ sku_id, tier, duration, performer_count, price, confidence, reasoning }] }

POST /v1/ai/recommend-price
  输入：{ sku_id, tier, duration, crowd_size, activity_type, location, urgency }

| 咖位/Tier | Tier | 艺人级别，T0（顶级）→ T6（新人） |
| 业务线 | Business Line | 脱口秀/即兴喜剧/漫才/魔术喜剧/亲子喜剧 |

| 运营 | Operator / Admin | 平台操作人员 |
| 经纪公司 | Agency | 管理多名艺人的演出经纪机构 |
| 独立艺人 | Independent Artist | 个人入驻、自己管理SKU的艺人 |
